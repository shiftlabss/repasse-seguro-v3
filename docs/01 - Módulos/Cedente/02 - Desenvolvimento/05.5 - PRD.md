## ⚙️ 1. Requisitos Não Funcionais (RNFs)

### 1.1 Performance

---

**RNF-001: Tempo de resposta da API — endpoints críticos**
- **Descrição:** Endpoints críticos (autenticação, aceite de proposta, criação de caso, upload de documento) devem responder em até 200ms no percentil 95 em condições normais de carga.
- **Métrica:** p95 latency ≤ 200ms medido via Sentry APM em produção.
- **Consequência se violado:** Degradação de experiência; possível timeout em integrações com ZapSign e parceiro Escrow.

---

**RNF-002: Tempo de resposta do Guardião do Retorno**
- **Descrição:** Primeira resposta do Guardião deve ser entregue via SSE em até 3 segundos. Resposta completa em até 10 segundos para queries complexas.
- **Métrica:** p95 time-to-first-token ≤ 3s medido via Langfuse.
- **Consequência se violado:** Experiência de chat degradada; abandono pelo Cedente.

---

**RNF-003: Tempo de carregamento das telas principais**
- **Descrição:** Dashboard, Meus Casos e Propostas devem carregar em até 2 segundos no First Contentful Paint (FCP) em conexão 4G.
- **Métrica:** FCP ≤ 2000ms medido via Core Web Vitals no Vercel Analytics.
- **Consequência se violado:** Taxa de abandono aumentada; impacto negativo no SEO (landing pages).

---

**RNF-004: Atualização em tempo real das notificações**
- **Descrição:** Badge de notificações no painel deve ser atualizado em até 30 segundos após qualquer evento via Supabase Realtime.
- **Métrica:** Latência de notificação ≤ 30s em 99% dos eventos. `[Origem: RN-057]`
- **Consequência se violado:** Cedente sem informação em tempo real sobre propostas — violação direta de RN-057.

---

**RNF-005: Upload de documentos**
- **Descrição:** Upload de arquivos até 10 MB deve ser concluído em até 30 segundos em conexão 4G.
- **Métrica:** p95 upload time ≤ 30s para arquivos de 10 MB.
- **Consequência se violado:** Cedente abandona o upload; dossiê nunca completo.

### 1.2 Disponibilidade e Resiliência

---

**RNF-006: Disponibilidade do serviço**
- **Descrição:** A plataforma deve ter disponibilidade de 99,5% medida mensalmente (uptime). Manutenções programadas devem ser comunicadas com 48h de antecedência.
- **Métrica:** Uptime ≥ 99,5% medido via Sentry ou ferramenta de monitoramento externo.
- **Consequência se violado:** Cedentes não conseguem acessar o painel; propostas podem expirar sem resposta; risco de perda de fechamentos.

---

**RNF-007: Tolerância a falhas de integrações externas**
- **Descrição:** Falhas nas integrações externas (ZapSign, Receita Federal, parceiro Escrow) não devem derrubar o sistema. Cada integração tem tratamento de fallback definido: ZapSign indisponível → mensagem amigável + retry; Receita Federal indisponível → modo de contingência com verificação manual pelo Admin; Escrow indisponível → exibe último status com timestamp.
- **Métrica:** Zero interrupções da plataforma por falha em integração externa.
- **Consequência se violado:** Falha em ZapSign derruba todo o fluxo de assinaturas.

---

**RNF-008: Recuperação após falha de sessão**
- **Descrição:** Em caso de falha de sessão ou queda de conexão, formulários em preenchimento devem ser preservados em rascunho local (localStorage/IndexedDB). Ao reconectar, dados são restaurados automaticamente.
- **Métrica:** Zero perdas de dados de formulário por falha de conexão.
- **Consequência se violado:** Cedente perde dados do wizard de cadastro — abandono do fluxo.

### 1.3 Segurança

---

**RNF-009: Isolamento de dados entre usuários**
- **Descrição:** Nenhum usuário pode acessar dados de outro usuário via API, URL ou manipulação de cliente. Row Level Security (RLS) no Supabase deve estar habilitado em todas as tabelas de domínio. Testes de isolamento fazem parte do pipeline de CI.
- **Métrica:** Zero incidentes de vazamento de dados entre Cedentes em produção. `[Origem: RN-011]`
- **Consequência se violado:** Violação de LGPD; risco jurídico e de reputação crítico.

---

**RNF-010: Autenticação e gestão de sessão**
- **Descrição:** Autenticação via JWT + Supabase Auth. Tokens com expiração de 24 horas de inatividade. Refresh tokens com rotação automática. Blacklist de tokens em caso de logout ou bloqueio.
- **Métrica:** Zero sessões válidas após logout ou bloqueio de conta.
- **Consequência se violado:** Sessões hijacked; acesso não autorizado a dados financeiros.

---

**RNF-011: Proteção contra ataques comuns**
- **Descrição:** Rate limiting em todos os endpoints de autenticação (máx. 5 req/min por IP). Headers de segurança via Helmet. HTTPS obrigatório em todos os ambientes. Validação de MIME type real em uploads. CSP configurado para prevenir XSS.
- **Métrica:** Zero vulnerabilidades críticas ou altas no OWASP Top 10 em audit semestral.
- **Consequência se violado:** Ataques de brute force, upload de malware, XSS.

---

**RNF-012: Conformidade com a LGPD**
- **Descrição:** Dados pessoais coletados devem ter base legal definida. Solicitação de exclusão processada em até 15 dias corridos. Logs de auditoria imutáveis com retenção mínima de 5 anos. Dados de Cessionários nunca expostos ao Cedente.
- **Métrica:** 100% das solicitações de exclusão processadas dentro do prazo. Zero incidentes de exposição de dados entre partes. `[Origem: RN-010, RN-011, RN-012]`
- **Consequência se violado:** Multas LGPD (até 2% do faturamento, limitado a R$ 50M), responsabilidade civil.

### 1.4 Escalabilidade

---

**RNF-013: Capacidade inicial de usuários concorrentes**
- **Descrição:** O sistema deve suportar 500 Cedentes ativos simultaneamente sem degradação de performance no lançamento. Escalabilidade horizontal deve ser possível via infraestrutura Railway (backend) e Vercel (frontend).
- **Métrica:** p95 latency ≤ 200ms com 500 usuários concorrentes em teste de carga.
- **Consequência se violado:** Sistema lento ou indisponível em picos de acesso.

---

**RNF-014: Crescimento de dados**
- **Descrição:** O banco de dados (PostgreSQL via Supabase) deve suportar 100.000 casos com dossiê completo (avg 10 documentos por caso) sem degradação de consultas. Índices obrigatórios em todas as colunas de filtro frequente (status do caso, ID do Cedente, data de criação).
- **Métrica:** Consultas de listagem de casos ≤ 100ms com 100.000 casos em produção.
- **Consequência se violado:** Telas de Meus Casos e Dashboard lentas com crescimento do produto.

### 1.5 Acessibilidade e Usabilidade

---

**RNF-015: Acessibilidade WCAG 2.1 AA**
- **Descrição:** Toda a interface web deve atender WCAG 2.1 nível AA: contraste mínimo 4.5:1, navegação por teclado, aria-labels, texto alternativo. `[Origem: RN-085]`
- **Métrica:** Zero violações críticas de WCAG 2.1 AA em auditoria automatizada (axe) e manual semestral.
- **Consequência se violado:** Exclusão de usuários com deficiência visual ou motora; risco de ação judicial por discriminação.

---

**RNF-016: Responsividade mobile**
- **Descrição:** Interface funcional e sem overflow em viewports de 320px a 1920px. Breakpoints obrigatórios: 320px, 768px, 1024px, 1280px.
- **Métrica:** Zero bugs de layout em testes nos breakpoints obrigatórios via Playwright.
- **Consequência se violado:** Experiência quebrada para Cedentes que usam apenas celular — maioria do público-alvo.

### 1.6 Observabilidade e Operação

---

**RNF-017: Logs estruturados em todas as camadas**
- **Descrição:** Backend NestJS usa Pino com logs estruturados JSON: request ID, timestamp, nível, contexto do usuário (ID mascarado), duração. Logs de auditoria separados dos logs de aplicação, com retenção mínima de 5 anos.
- **Métrica:** 100% das requisições com request ID rastreável do frontend ao banco.
- **Consequência se violado:** Impossibilidade de rastrear bugs e incidentes em produção.

---

**RNF-018: Monitoramento de erros em tempo real**
- **Descrição:** Sentry obrigatório em todas as camadas (Next.js, NestJS, Expo). Alertas configurados para taxa de erros > 1% nas últimas 1h. LLM costs e latências monitorados via Langfuse com alertas para desvios > 50% da baseline.
- **Métrica:** Alerta disparado em até 5 minutos após qualquer degradação crítica.
- **Consequência se violado:** Incidentes não detectados até que Cedentes reclamem.

---

**RNF-019: Analytics de produto obrigatório desde o lançamento**
- **Descrição:** PostHog instalado em web e mobile desde o dia 1. Funil de conversão cadastro → dossiê → triagem → proposta → fechamento monitorado. Eventos críticos rastreados: cadastro concluído, upload de documento, proposta recebida, proposta aceita, fechamento.
- **Métrica:** Dados de funil disponíveis a partir do D+1 do lançamento.
- **Consequência se violado:** Impossibilidade de otimizar o funil de conversão.

### 1.7 Conformidade Técnica com a Stack

---

**RNF-020: Stack obrigatória — zero desvios sem ADR**
- **Descrição:** Toda a implementação deve seguir o ShiftLabs Stacks v7.0 conforme definido no Doc 02 — Stacks. Qualquer desvio requer ADR documentado e aprovação do Tech Lead. Stack obrigatória: NestJS 10+, Prisma 6+, Supabase (PostgreSQL 17+), Redis/Upstash, RabbitMQ/CloudAMQP, Next.js 15, React Native 0.76 / Expo SDK 52, shadcn/ui, Tailwind CSS.
- **Métrica:** Zero dependências fora da stack aprovada em produção sem ADR correspondente.
- **Consequência se violado:** Inconsistência técnica entre módulos do produto; retrabalho na integração.

---

**RNF-021: TypeScript strict mode obrigatório**
- **Descrição:** `strict: true` obrigatório em todos os projetos TypeScript (backend, frontend, mobile). Arquivos `.js` proibidos em código de produção.
- **Métrica:** Zero erros de TypeScript no build de CI com strict mode ativo.
- **Consequência se violado:** Bugs de tipo em produção; inconsistências de dados.

---

**RNF-022: Cobertura de testes mínima**
- **Descrição:** Cobertura de testes unitários ≥ 80% em módulos de domínio crítico (autenticação, estados do caso, cálculo de comissão, validações de documentos). Testes E2E obrigatórios para o fluxo feliz completo (cadastro → fechamento).
- **Métrica:** Coverage report ≥ 80% em CI para módulos críticos; todos os testes E2E passando antes de merge para main.
- **Consequência se violado:** Regressões em produção nos fluxos de receita.
