## 1. Módulo: Financeiro (Gestão)

### RF-075 — Monitoramento de Contas Escrow em tempo real

**Origem:** `Conforme RN-087 (Parte 01.4)`

O sistema deve exibir lista de todas as Contas Escrow com status atualizado em tempo real no módulo Financeiro.

**Critério de aceite:**
- Tabela: ID do caso, Status da Conta Escrow, Valor Esperado, Valor Depositado, Data do Depósito, Data Prevista de Distribuição, Ações.
- Filtros como chips clicáveis: Aberta, Depósito Confirmado, Em Distribuição, Distribuída, Congelada, Estornada. Múltiplos filtros combinados.
- Destaque visual com label textual: "Prazo próximo" (amarelo, ≤3 dias) / "Prazo estourado" (vermelho).
- Clique em conta: drawer lateral com breakdown detalhado (Cedente vs. RS como barras empilhadas), histórico de movimentações, dados do caso.
- Atualização em tempo real via Supabase Realtime (polling de 30s como fallback).

---

### RF-076 — Bloqueio de distribuição com aprovação do Master e escalonamento

**Origem:** `Conforme RN-088 (Parte 01.4)`, `Conforme RN-088.a (Parte 01.4)`

O sistema deve permitir ao Gestor bloquear distribuições com aprovação do Master e escalonamento automático.

**Critério de aceite:**
- "Bloquear Distribuição": justificativa obrigatória (mín. 20 caracteres). Solicitação enviada ao Master.
- Notificação imediata ao Master (e-mail + painel).
- 8h sem resposta: reenvio com urgência.
- 16h sem resposta: todos os Masters ativos notificados.
- 24h sem resposta do Master: distribuição retomada automaticamente. Incidente registrado no log com severidade alta.
- Master deve registrar justificativa da não-resposta em até 2 dias úteis. Incidentes não justificados aparecem no Relatório de Auditoria mensal.
- Bloqueio aprovado: distribuição suspensa até Gestor resolver irregularidade e liberar manualmente.

### RF-077 — Processamento de estorno com modal de confirmação

**Origem:** `Conforme RN-089 (Parte 01.4)`

O sistema deve exibir modal de confirmação antes de processar estorno, com geração automática de comprovante.

**Critério de aceite:**
- "Iniciar Reversão": modal com resumo (valor depositado, partes envolvidas).
- Confirmação: Conta Escrow congelada imediatamente. Fluxo de reversão iniciado.
- Estorno sempre integral (sem estorno parcial — conforme RF-008).
- Comprovante gerado automaticamente e salvo no dossiê do caso.
- Celcoin API chamada para executar estorno financeiro (RF-038).

### RF-078 — Painel de inadimplência com alertas de prazo e ações por perfil

**Origem:** `Conforme RN-090 (Parte 01.4)`

O sistema deve exibir painel de inadimplência com filtros de prazo e ações diferenciadas por perfil.

**Critério de aceite:**
- Lista: casos com depósito pendente, dias desde envio das instruções, prazo restante, número de prorrogações concedidas.
- "Prazo próximo" (amarelo) ≤ 3 dias úteis. "Prazo estourado" (vermelho) prazo ultrapassado. Labels textuais (não apenas cor).
- Gestor Financeiro: "Enviar Lembrete" e "Marcar para Cancelamento".
- Coordenador: "Conceder Prorrogação" (máximo 1 vez, +5 dias úteis).
- Exportação CSV da lista de inadimplência (RN-149).
- QA: Analista não deve ver o botão "Marcar para Cancelamento". Testar acesso por perfil.

### RF-079 — Conciliação bancária com detecção de divergências

**Origem:** `Conforme RN-091 (Parte 01.4)`

O sistema deve comparar valores confirmados pelo parceiro financeiro com registros internos e alertar divergências.

**Critério de aceite:**
- Valor confirmado diferença ≤ R$0,50: badge "Conciliado" (verde com label textual).
- Diferença > R$0,50: badge "Divergência detectada" (vermelho com label textual) com: valor esperado, valor confirmado, diferença.
- Movimentação com divergência: status "Em conciliação". Gestor deve investigar e registrar justificativa antes de fechar.
- QA: testar divergência de R$1,00 e verificar que status muda para "Em conciliação".

### RF-080 — Relatório financeiro mensal automático

**Origem:** `Conforme RN-092 (Parte 01.4)`

O sistema deve gerar e enviar automaticamente relatório financeiro mensal no 1º dia útil de cada mês.

**Critério de aceite:**
- Job agendado: 1º dia útil de cada mês às 07h00 (America/Fortaleza) `[DECISÃO AUTÔNOMA: 07h00 para Gestor receber antes do início do expediente]`.
- Relatório contém: receita total, volume de transações, ticket médio, taxa de reversão, projeção de receita baseada no pipeline.
- Enviado por e-mail ao Gestor Financeiro e ao Master em PDF e CSV.
- Relatório também exportável manualmente a qualquer momento no menu Relatórios.
- QA: testar que job executa no 1º dia útil (e não no 1º dia do mês se cair no fim de semana).

## 2. Módulo: Supervisão IA

### RF-081 — Dashboard de supervisão IA com refresh de 10 segundos

**Origem:** `Conforme RN-093 (Parte 01.4)`

O sistema deve exibir painel de métricas dos agentes de IA com log atualizado em até 10 segundos.

**Critério de aceite:**
- Dashboard: cards superiores (ações 24h, confiança média, alertas pendentes, takeovers ativos).
- Log de ações: tabela com scroll. Colunas: timestamp, agente, caso, tipo de ação, confiança (badge colorido), resultado.
- Confiança ≥80%: badge verde "Alta". 60-79%: badge amarelo "Média". <60%: badge vermelho "Baixa".
- Log atualiza em até 10 segundos via Supabase Realtime.
- Aba "Alertas" com badge numérico separada da aba "Log".

---

### RF-082 — Alertas de baixa confiança com ações do Coordenador

**Origem:** `Conforme RN-094 (Parte 01.4)`

O sistema deve gerar alertas automáticos para ações abaixo do limiar de confiança com 3 opções de resposta.

**Critério de aceite:**
- Ação abaixo do limiar: alerta gerado automaticamente. Badge vermelho na aba "Alertas" + ícone no menu da sidebar.
- Alerta exibe: ação executada, confiança obtida vs. limiar, dados de entrada, resultado.
- 3 botões: "Confirmar" (verde), "Reverter" (amarelo), "Takeover" (vermelho).
- Cada ação exige justificativa escrita (mín. 20 caracteres) `[DECISÃO AUTÔNOMA: 20 chars para garantir justificativa mínima]`.
- Justificativa gera registro imutável no audit trail.
- QA: ação sem justificativa deve retornar erro de validação.

---

### RF-083 — Takeover manual com confirmação e limite de 5 simultâneos

**Origem:** `Conforme RN-095 (Parte 01.4)`

O sistema deve permitir takeover manual de agentes com modal de confirmação e limite de 5 simultâneos por operador.

**Critério de aceite:**
- "Assumir Controle": modal com nome do agente, caso afetado, última ação, campo de justificativa obrigatório.
- Takeover ativo: agente pausado apenas para o caso específico. Demais casos continuam normais.
- Barra superior: indicador "X takeover(s) ativo(s)" com link para o painel.
- Contador no painel: "X/5 takeovers em uso."
- 5 takeovers simultâneos atingidos + nova falha crítica: caso escalado automaticamente ao Master.
- "Devolver Controle": modal de confirmação. Log de takeover registrado com duração e ações manuais realizadas.
- QA: testar limit de 6 takeovers — 6ª tentativa deve resultar em escalação ao Master.

---

### RF-084 — Takeover automático por falha crítica (6 eventos gatilho)

**Origem:** `Conforme RN-096 (Parte 01.4)`

O sistema deve executar takeover automático imediato em 6 eventos de falha crítica do agente.

**Critério de aceite:**
- Loop (3+ ações iguais em 60s): takeover automático imediato.
- Timeout de ação (>30s): takeover automático imediato.
- Erro de sistema (falha interna não tratada): takeover automático.
- Dados corrompidos (campos vazios ou valores fora de domínio): takeover automático.
- Conflito de versão (lock otimista — RN-130): takeover automático. [CORRIGIDO: PROBLEMA-009]
- Violação de regra de negócio (ex: aprovação de proposta abaixo do mínimo): takeover automático.
- Após takeover automático: agente não pode se reativar sozinho. Somente operador pode devolver controle após inspeção.
- Alerta crítico enviado imediatamente ao Coordenador e Master.
- QA: simular timeout e verificar takeover + alerta.

### RF-085 — Thresholds de confiança configuráveis com histórico de calibração

**Origem:** `Conforme RN-095.a (Parte 01.4)`, `Conforme RN-100 (Parte 01.4)`

O sistema deve permitir ao Master configurar thresholds de autonomia dos agentes com registro histórico de calibrações.

**Critério de aceite:**
- Thresholds configuráveis por agente (Guardião do Retorno e Analista de Oportunidades) em Configurações → IA.
- Intervalo permitido: 50% a 95%. Padrão inicial: 80%.
- Valor fora do intervalo: erro de validação inline.
- Alteração: registrada imediatamente no log de auditoria com: valor anterior, novo valor, timestamp, Master, justificativa.
- Histórico de calibrações visível para Coordenador e Master na tela de Supervisão IA.
- Alerta ao Master se taxa de reversão > 30% em 7 dias.

---

### RF-086 — Revisão diária obrigatória com escalonamento por inatividade

**Origem:** `Conforme RN-097 (Parte 01.4)`

O sistema deve monitorar o acesso diário do Coordenador à Supervisão IA e escalar alertas pendentes após inatividade.

**Critério de aceite:**
- 24h sem acesso: alerta ao Master informando inatividade do Coordenador e número de alertas pendentes.
- 48h sem acesso: notificação prioritária ao Master. Todos os alertas pendentes escalados ao Master automaticamente.
- 72h sem acesso: incidente de governança registrado no audit trail. Master deve tomar ação.
- Monitoramento individual por Coordenador. Múltiplos Coordenadores monitorados de forma independente.

---

### RF-087 — Escalação automática de alertas não resolvidos após 48h

**Origem:** `Conforme RN-098 (Parte 01.4)`

O sistema deve escalar ao Master alertas de IA não resolvidos após 48 horas.

**Critério de aceite:**
- Alerta pendente por 48h: escalado automaticamente ao Master com badge "Escalado automaticamente".
- Master vê mesmo conjunto de ações disponíveis (Confirmar/Reverter/Takeover).
- Alerta escalado: badge amarelo + label "Escalado após 48h" no log.

---

### RF-088 — Modo Supervisão Total: confirmação obrigatória de todas as ações dos agentes

**Origem:** `Conforme RN-099 (Parte 01.4)`

O sistema deve suportar modo Supervisão Total onde todas as ações dos agentes aguardam confirmação humana.

**Critério de aceite:**
- Master ativa Supervisão Total em Configurações → IA.
- Neste modo: toda ação dos agentes entra em fila aguardando confirmação.
- 4h sem confirmação: alerta ao Coordenador.
- 8h sem confirmação: alerta ao Master com prioridade alta.
- 12h sem confirmação: ação descartada automaticamente. Caso recebe flag "Ação pendente manual". Analista notificado para executar manualmente. Evento registrado como "Timeout de supervisão" no audit trail.
- QA: ativar modo e verificar que nenhuma ação do agente é executada sem confirmação humana.

## 3. Módulo: Relatórios

### RF-089 — Relatório de SLA com métricas por etapa e Analista

**Origem:** `Conforme RN-100 (Parte 01.4)`, `Conforme RN-099 (Parte 01.4)`, `Conforme RN-107 (Parte 01.4)` [CORRIGIDO: PROBLEMA-007] [DECISÃO APLICADA: DEC-004]

O sistema deve gerar Relatório de SLA com métricas detalhadas por etapa e por Analista responsável, com matriz de permissões de acesso por perfil aplicada a todo o módulo Relatórios.

**Critério de aceite:**
- Métricas por etapa: % casos dentro do SLA, % em atraso, tempo médio de permanência em cada estado.
- Métricas por Analista: casos gerenciados, SLAs cumpridos vs. estourados, tempo médio por etapa.
- Filtros: período, módulo, Analista, status do SLA.
- Gráficos com tabela alternativa acessível para leitores de tela (conforme RN-108).
- Exportação CSV e PDF com timestamp de geração.

**Matriz de permissões de acesso ao módulo Relatórios (RN-107):**

| **Perfil** | **Relatórios acessíveis** | **Bloqueados** |
|---|---|---|
| Analista | Nenhum (sem acesso ao módulo) | Todos |
| Coordenador | SLA, Volume, Conversão, Agentes IA | Receita, Auditoria |
| Gestor Financeiro | Receita, Auditoria | SLA, Volume, Conversão, Agentes IA |
| Master | Todos | — |

- QA: Analista GET /reports/sla → HTTP 403. Coordenador GET /reports/receita → HTTP 403. Coordenador GET /reports/auditoria → HTTP 403. Gestor Financeiro GET /reports/sla → HTTP 403. Gestor Financeiro GET /reports/volume → HTTP 403. Todos validados via guard RBAC no backend.

### RF-095 — Exportação de relatórios em CSV e PDF, e agendamento automático

**Origem:** `Conforme RN-106 (Parte 01.4)`, `Conforme RN-108 (agendamento) (Parte 01.4)` [CORRIGIDO: PROBLEMA-008] [DECISÃO APLICADA: DEC-005]

O sistema deve disponibilizar exportação de todos os relatórios em CSV e PDF com filtros aplicados, e agendamento automático de envio pelo Master.

**Critério de aceite:**
- Botão "Exportar CSV" e "Exportar PDF" disponíveis em todos os relatórios.
- Exportação gerada de forma assíncrona (job de background) para relatórios com > 1.000 registros.
- Notificação ao usuário quando exportação estiver pronta com link de download (válido por 24h).
- PDF formatado com logo do Repasse Seguro, título do relatório, período, filtros aplicados e timestamp de geração.
- **Agendamento de envio automático (apenas Master — RN-108):** Master acessa Configurações → Relatórios Agendados. Por relatório: selecionar frequência (diária, semanal, mensal), destinatários (e-mails), formato (CSV ou PDF). Job Cron gerenciado via Bull/BullMQ (RabbitMQ). Cancelamento de agendamento disponível a qualquer momento. Relatório enviado automaticamente por e-mail no horário configurado com os filtros ativos no momento de criação do agendamento.
- QA: exportar relatório de Auditoria com 500+ registros e verificar integridade dos dados. Configurar agendamento semanal → verificar que e-mail é enviado na semana seguinte com conteúdo correto.

## 4. Módulo: Configurações

### RF-097 — Parâmetros de comissão configuráveis pelo Master

**Origem:** `Conforme RN-109 (Parte 01.4)`, `Conforme RN-119 (Parte 01.4)`, `Conforme RN-120 (Parte 01.4)`

O sistema deve permitir ao Master configurar os parâmetros de comissão sem deploy de código.

**Critério de aceite:**
- Configurações → Comissões: campos editáveis pelo Master: % Distrato Referência (padrão 50%), % Comissão RS (padrão 20%, intervalo 5% a 50%), Limiar de revisão do Delta (padrão R$100.000).
- Pré-visualização obrigatória antes de salvar: "Com esses parâmetros, um caso com Valor Recuperado de R$X teria comissão de R$Y."
- Alterações afetam apenas novos casos. Casos existentes usam parâmetros do snapshot criado em sua data de abertura (RN-111).
- Log de alteração registrado automaticamente no audit trail com: parâmetro, valor anterior, novo valor, timestamp, Master.
- QA: criar caso → alterar parâmetros → verificar que caso existente usa valores anteriores.

### RF-098 — Snapshot de configuração por caso na criação

**Origem:** `Conforme RN-111 (Parte 01.4)`

O sistema deve salvar um snapshot das configurações vigentes no momento de criação de cada caso.

**Critério de aceite:**
- Ao criar um caso: sistema copia as configurações ativas (comissões, SLAs, thresholds) para o registro do caso em campo JSON imutável.
- Cálculos do caso (comissão, SLA alerts) sempre usam o snapshot, nunca as configurações atuais.
- QA: criar caso → alterar % comissão → chegar no Fechamento → verificar que comissão usa valor do snapshot.

### RF-099 — Templates de comunicação editáveis pelo Master

**Origem:** `Conforme RN-112 (Parte 01.4)`

O sistema deve permitir ao Master editar os templates de mensagem usados nas notificações automáticas.

**Critério de aceite:**

- Editor WYSIWYG com suporte a variáveis: `{{nome_cedente}}`, `{{id_caso}}`, `{{valor}}`, `{{prazo}}`, etc.
- Preview do template renderizado antes de salvar.
- Alterações entram em vigor imediatamente para notificações futuras. Notificações já enviadas não são afetadas.
- Template original (padrão de fábrica) pode ser restaurado com 1 clique.

### RF-101 — Gestão de chaves de API de integrações externas

**Origem:** `Conforme RN-114 (Parte 01.4)`

O sistema deve permitir ao Master configurar chaves de API das integrações externas com mascaramento de segurança.

**Critério de aceite:**
- Configurações → Integrações: campos para ZapSign API Key, ZapSign Webhook Secret, Celcoin API Key, Meta Cloud API token, Twilio SID/Token.
- Valores exibidos mascarados após salvar (ex: `sk_live_****1234`). Nunca expostos em texto completo.
- Botão "Testar conexão" para cada integração: verifica conectividade e autenticação. Exibe: ✅ "Conexão bem-sucedida" ou ❌ "Falha: [motivo]".
- Webhook Secret ZapSign e Celcoin: usados para validação HMAC-SHA256 (RF-026, RF-037).

### RF-102 — Pré-visualização obrigatória antes de salvar configurações

**Origem:** `Conforme RN-118 (Parte 01.4)`

O sistema deve exigir pré-visualização de impacto antes de salvar qualquer alteração de configuração.

**Critério de aceite:**
- Configurações alteradas mas não salvas: botão "Salvar" desabilitado até o Master clicar em "Pré-visualizar impacto".
- Pré-visualização: mostra exemplo concreto do impacto da mudança (ex: "Com 25% de comissão, um caso com base de R$100.000 geraria R$25.000 de comissão").
- Após pré-visualização: botão "Salvar" habilitado. Master pode prosseguir ou cancelar.
- QA: tentar salvar configuração sem pré-visualizar → botão deve permanecer desabilitado.

### RF-103 — Logs de alteração de parâmetros no audit trail

**Origem:** `Conforme RN-116 (Parte 01.4)`

O sistema deve registrar no audit trail toda alteração de parâmetro de configuração.

**Critério de aceite:**
- Cada alteração: parâmetro, valor anterior, novo valor, timestamp, Master responsável, motivo (se configurações críticas, campo obrigatório).
- Log acessível no Relatório de Auditoria (RF-093).
- Nenhum registro pode ser deletado ou editado do audit trail.
- QA: alterar um parâmetro e verificar entry no audit trail com todos os campos.

## 5. Módulo: Transversais (parte administrativa)

### RF-104 — Audit trail append-only com retenção de 5 anos e registro de transições de status

**Origem:** `Conforme RN-129 (Parte 01.5)`, `Conforme RN-016 (Parte 01.1)` [CORRIGIDO: PROBLEMA-003, PROBLEMA-010] [DECISÃO APLICADA: DEC-007]

O sistema deve manter audit trail imutável de todas as ações com retenção de 5 anos em schema separado, incluindo registro automático obrigatório de cada transição de status de caso.

**Critério de aceite:**
- Toda ação crítica registrada: quem, o quê, quando, qual objeto, estado anterior e posterior.
- **Transições de status de caso (RN-016):** toda mudança de status de caso registra automaticamente e de forma imutável: ID do caso, status anterior, novo status, data e hora (timestamp UTC), nome do operador responsável e motivo (quando aplicável). Este registro é automático — não pode ser omitido por nenhum código path. Nenhum perfil (incluindo Master) pode editar ou excluir registros de transição de status.
- Schema separado append-only no banco. Nenhuma mutation (UPDATE/DELETE) permitida neste schema via Prisma.
- Retenção: 5 anos mínimos por exigência legal.
- QA (RN-016 — T30): executar mudança de status de caso → verificar que registro de auditoria contém todos os 5+ campos obrigatórios. QA (RN-016 — T31): tentativa de editar ou deletar registro via API → HTTP 403. QA geral: tentativa de DELETE em tabela do audit trail via Prisma → deve falhar em runtime por RLS.

### RF-105 — Lock otimista para edições concorrentes

**Origem:** `Conforme RN-130 (Parte 01.5)`

O sistema deve implementar lock otimista para evitar sobrescrita de dados em edições concorrentes.

**Critério de aceite:**
- Toda entidade editável tem campo `version Int`. Ao editar, enviar `version` atual. Backend verifica: se versão do banco ≠ versão enviada → HTTP 409 com mensagem "Conflito: o registro foi alterado por outro usuário. Recarregue e tente novamente."
- Frontend: ao receber 409, exibe toast com botão "Recarregar dados" que refaz GET do recurso.
- QA: abrir mesmo caso em 2 abas. Editar na aba 1, salvar. Na aba 2, editar e salvar → deve retornar 409.

---

### RF-106 — Idempotency key em todas as ações de mutação

**Origem:** `Conforme RN-133 (Parte 01.5)`

O sistema deve garantir idempotência em todas as ações de mutação crítica via header `Idempotency-Key`.

**Critério de aceite:**
- Header `Idempotency-Key` obrigatório em: transições de estado, propostas, Fechamento, distribuição escrow, estorno.
- Mesmo `Idempotency-Key` + mesmo endpoint: segunda requisição retorna a mesma resposta da primeira sem reprocessar.
- Chaves armazenadas no Redis com TTL de 24 horas.
- QA: submeter proposta com mesma idempotency key 2x → segunda deve retornar resposta idêntica à primeira sem criar proposta duplicada.

---

### RF-107 — Webhooks de saída com retry e dead-letter queue

**Origem:** `Conforme RN-138 (Parte 01.5)`

O sistema deve suportar webhooks de saída com retry automático e dead-letter queue.

**Critério de aceite:**
- Webhook de saída com falha (HTTP 4xx/5xx ou timeout): retry automático com backoff exponencial (1s, 2s, 4s, 8s, 16s).
- Após 5 tentativas sem sucesso: evento movido para dead-letter queue (DLQ). Master notificado.
- DLQ acessível em Configurações → Integrações → "Fila de reenvio". Master pode reprocessar manualmente.
- Todos os webhooks enviados com HMAC-SHA256 para validação pelo receptor.
- QA: configurar endpoint de teste que retorna 500. Verificar retry + DLQ após 5 tentativas.

---

### RF-108 — Retry automático com backoff exponencial para integrações externas

**Origem:** `Conforme RN-150 (Parte 01.5)`

O sistema deve implementar retry automático com backoff exponencial para todas as chamadas a APIs externas.

**Critério de aceite:**
- Falha em API externa (ZapSign, Celcoin, Meta, Twilio): retry automático com backoff exponencial.
- Configuração: máximo 3 tentativas, delays de 1s/5s/15s `[DECISÃO AUTÔNOMA: 3 tentativas com delays progressivos, balanceando resiliência sem sobrecarregar o sistema externo]`.
- Após 3 tentativas: falha registrada no audit trail. Evento movido para DLQ. Operador relevante notificado.
- Cada tentativa registrada no log com: timestamp, endpoint, HTTP status, duração.
