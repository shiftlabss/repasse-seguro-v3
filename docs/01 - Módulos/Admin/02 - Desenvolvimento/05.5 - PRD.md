## 1. Requisitos Não Funcionais

### RNF-001 — Performance: Tempo de resposta da API

**Origem:** `[DECISÃO AUTÔNOMA: benchmarks de plataforma operacional financeira]`

O sistema deve responder às requisições da API dentro dos limites estabelecidos sob carga normal de operação.

**Critério de aceite:**
- p50 < 200ms para leituras (GET).
- p95 < 500ms para leituras (GET).
- p95 < 1.000ms para escritas (POST/PUT/PATCH) em operações não financeiras.
- p95 < 2.000ms para operações financeiras (Fechamento, distribuição, estorno).
- Medido em produção via Sentry Performance + PostHog.

---

### RNF-002 — Performance: Carregamento da interface web

**Origem:** `Conforme RN-011 (Parte 01.1)`

O sistema deve carregar as telas principais dentro dos limites de tempo definidos.

**Critério de aceite:**
- Dashboard (com KPIs): FCP < 1.5s, LCP < 2.5s.
- Pipeline (kanban): interativo em < 3s com até 200 casos visíveis.
- Skeleton screens obrigatórios durante carregamento. Spinners genéricos proibidos.
- Bundle JavaScript total gzipped < 300KB.
- Rollup Plugin Visualizer usado em CI para alertar se bundle exceder o limite.

---

### RNF-003 — Disponibilidade e SLA de infraestrutura

**Origem:** `[DECISÃO AUTÔNOMA: nível adequado para operação financeira sem 24/7 crítico]`

O sistema deve manter disponibilidade de 99,5% medida mensalmente.

**Critério de aceite:**
- Uptime Railway (backend) e Vercel (frontend): monitorados via Uptime Robot ou similar.
- Downtime planejado (manutenção): notificado com 24h de antecedência ao Master.
- Metas: 99,5% uptime mensal = máximo 3,6 horas de downtime por mês.
- Incident response: equipe notificada em até 5 minutos de downtime detectado.

---

### RNF-004 — Escalabilidade: Operações simultâneas

**Origem:** `[DECISÃO AUTÔNOMA: estimativa para operação inicial e crescimento]`

O sistema deve suportar as operações simultâneas previstas para o MVP e o crescimento inicial.

**Critério de aceite:**
- MVP: suporte a 10 operadores simultâneos sem degradação de performance.
- Crescimento: arquitetura stateless no backend (NestJS) permite escalar horizontalmente sem redesign.
- Redis para cache de sessões e blacklist de tokens: suporte a 1.000+ sessões ativas.
- Supabase Realtime: suporte a 50+ subscriptions simultâneas por instância.

---

### RNF-005 — Segurança: Autenticação e autorização

**Origem:** `Conforme RN-004, RN-005, RN-006, RN-007 (Parte 01.1)`, `Conforme RN-134 (Parte 01.5)`

O sistema deve implementar autenticação robusta com 2FA e controle de acesso por perfil.

**Critério de aceite:**
- 2FA TOTP obrigatório para 100% dos operadores. Sem opção de bypass.
- JWT: access token em memória (Zustand), refresh token em httpOnly cookie (web) / SecureStore (mobile).
- Rate limiting: login 5 tentativas / 30 minutos por IP e por conta.
- RBAC: Analista não acessa Financeiro. Gestor não acessa Triagem. Coordenador não acessa Configurações.
- Supabase RLS habilitado em todas as tabelas com dados de usuários.
- CORS configurado explicitamente. `Access-Control-Allow-Origin: *` proibido em produção.
- Pen test de RBAC: testar todos os 4 perfis × todos os 10 módulos.

---

### RNF-006 — Segurança: HTTPS e headers

**Origem:** `Doc 02 Stacks, seção 9`

O sistema deve usar HTTPS em todos os ambientes de produção e staging com headers de segurança corretos.

**Critério de aceite:**
- HTTPS (TLS 1.2+) obrigatório. HTTP proibido em produção. Redirecionamento automático para HTTPS.
- Helmet.js configurado com: CSP, X-Frame-Options, X-Content-Type-Options, HSTS.
- CSRF protection obrigatório para endpoints que usam cookies httpOnly.
- Deploy sem HTTPS bloqueado no CI.

---

### RNF-007 — Segurança: LGPD e proteção de dados

**Origem:** `Conforme RN-137 (Parte 01.5)`, `Conforme RN-070, RN-071 (Parte 01.3)`

O sistema deve estar em conformidade com a LGPD para todos os dados pessoais processados.

**Critério de aceite:**
- Base legal documentada para cada categoria de dado (identificação, contato, financeiro, comportamental, documental).
- CPF/CNPJ mascarado no backend para Analistas (RF-058).
- Exportação de usuários sem CPF/CNPJ.
- Solicitações de exclusão atendidas em até 15 dias corridos.
- Dados vinculados a casos ativos/concluídos: retidos por 5 anos (obrigação legal).
- Consentimento e opt-out implementados.

---

### RNF-008 — Acessibilidade: WCAG AA

**Origem:** `Conforme RN-059 (Parte 01.3)`, `Conforme RN-108 (Parte 01.4)`, Doc 02 Stacks seção 3.5

O sistema deve atingir conformidade WCAG 2.1 Nível AA em todas as telas.

**Critério de aceite:**
- Contraste mínimo 4.5:1 para texto normal, 3:1 para texto grande.
- Alertas de SLA não dependem apenas de cor: ícone diferenciado + label textual obrigatórios.
- Gráficos com tabela alternativa (RF-096).
- Navegação completa por teclado (Tab, Shift+Tab, Enter, Escape) em todos os modais e dropdowns.
- Tamanho mínimo de toque: 44×44px.
- `aria-label` descritivo em todos os elementos interativos sem texto visível.
- Testado com VoiceOver (macOS) e Narrator (Windows) nas telas críticas.

---

### RNF-009 — Observabilidade: Logs estruturados

**Origem:** `Doc 02 Stacks, seção 2.1 (Pino)`

O sistema deve produzir logs estruturados em JSON para todas as requisições e erros.

**Critério de aceite:**
- Pino configurado em todos os serviços NestJS. Campos obrigatórios: request_id, timestamp, level, service, method, path, status_code, duration_ms, user_id.
- Logs de erro: stack trace + contexto da requisição.
- Deploy sem logs estruturados bloqueado no CI.
- Nenhum `console.log` em código de produção.

---

### RNF-010 — Observabilidade: Error tracking

**Origem:** `Doc 02 Stacks, seção 2.1 (@sentry/nestjs, @sentry/react)`

O sistema deve capturar e rastrear erros em produção via Sentry em todas as camadas.

**Critério de aceite:**
- Sentry configurado no backend (NestJS), frontend (React), mobile (React Native).
- Source maps enviados no build de produção.
- Error Boundary global no frontend: erros não derrubam toda a UI.
- Alertas configurados para: errors críticos (HTTP 5xx), performance degradada (p95 acima do threshold), taxa de erro acima de 1%.
- Deploy sem Sentry configurado bloqueado no CI.

---

### RNF-011 — Observabilidade: Analytics e session replay

**Origem:** `Doc 02 Stacks, seção 3.2 (PostHog)`

O sistema deve capturar eventos de uso e session replay para debugging e melhoria de produto.

**Critério de aceite:**
- PostHog configurado no frontend e mobile.
- Wrapper centralizado em `services/analytics.ts`.
- Eventos rastreados: login, navegação entre módulos, ações críticas (fechar caso, aceitar proposta, confirmar fechamento).
- Session replay habilitado para debugging. Dados sensíveis mascarados no replay.
- Feature flags disponíveis para rollout gradual de funcionalidades.

---

### RNF-012 — Qualidade de código: TypeScript strict

**Origem:** `Doc 02 Stacks, seção 2.1`

Todo código deve usar TypeScript com `strict: true` sem exceções.

**Critério de aceite:**
- `tsconfig.json` com `strict: true` em todos os packages do monorepo.
- Nenhum arquivo `.js` ou `.jsx` no projeto.
- `any` proibido exceto com comentário justificado `// eslint-disable-next-line @typescript-eslint/no-explicit-any`.
- CI bloqueia build se `tsc --noEmit` retornar erros.

---

### RNF-013 — Qualidade de código: Linting e formatação

**Origem:** `Doc 02 Stacks, seção 6`

Todo código deve passar por ESLint e Prettier antes de cada commit.

**Critério de aceite:**
- Husky + lint-staged: ESLint + Prettier executados no pre-commit.
- Configurações: `semi: true`, `singleQuote: true`, `trailingComma: 'all'`, `printWidth: 100`.
- CI falha se lint retornar erros.

---

### RNF-014 — Qualidade de código: Testes de integração para endpoints críticos

**Origem:** `Doc 02 Stacks, seção 7`

Os endpoints críticos devem ter testes de integração via Supertest + NestJS Testing Module.

**Critério de aceite:**
- Endpoints críticos com cobertura obrigatória: auth (login, 2FA, refresh), transições de estado do caso, Fechamento, distribuição escrow, propostas, RBAC.
- Testes de integração executados no CI antes de cada deploy.
- Falha em teste de integração bloqueia deploy.

---

### RNF-015 — Confiabilidade: Idempotência em operações críticas

**Origem:** `Conforme RN-133 (Parte 01.5)`

Todas as operações de mutação crítica devem ser idempotentes via `Idempotency-Key`.

**Critério de aceite:**
- Conforme RF-106.
- Testado em CI com cenários de duplo submit.

---

### RNF-016 — Confiabilidade: Soft delete em entidades de domínio

**Origem:** `Conforme RN-135 (Parte 01.5)`, Doc 02 Stacks seção 2.3

Entidades de domínio nunca são hard deleted.

**Critério de aceite:**
- Campos `deleted_at DateTime? @db.Timestamptz` em: casos, usuários, propostas, documentos, envelopes ZapSign.
- Queries padrão filtram `deleted_at IS NULL` automaticamente via Prisma middleware.
- DELETE físico proibido nas tabelas de domínio via RLS no Supabase.

---

### RNF-017 — Confiabilidade: Timestamps em UTC

**Origem:** `Conforme RN-136 (Parte 01.5)`, Doc 02 Stacks seção 2.3

Todos os timestamps armazenados em UTC. Exibição convertida para timezone do operador.

**Critério de aceite:**
- `@db.Timestamptz` obrigatório em todos os campos de data/hora no schema Prisma.
- `@db.Timestamp` sem timezone proibido.
- Frontend usa `date-fns-tz` para conversão de UTC para timezone do operador.
- QA: gravar evento com operador em UTC-3, verificar que banco armazena em UTC, verificar que interface exibe hora local.

---

### RNF-018 — Confiabilidade: Backup e disaster recovery

**Origem:** `[DECISÃO AUTÔNOMA: requisito implícito para plataforma financeira]`

O sistema deve ter backup automático e plano de disaster recovery documentado.

**Critério de aceite:**
- Supabase: backup automático diário habilitado. Ponto de recuperação: máximo 24 horas de perda de dados.
- Railway: deploy com rollback imediato disponível (último deploy estável).
- Procedure de DR documentada: como restaurar banco, como reverter deploy, contatos de emergência.
- Testado semestralmente.

---

### RNF-019 — Suportabilidade: Navegadores suportados

**Origem:** `[DECISÃO AUTÔNOMA: operadores em ambiente corporativo]`

O sistema deve suportar os navegadores mais usados em ambientes corporativos.

**Critério de aceite:**
- Chrome 120+, Firefox 120+, Edge 120+, Safari 17+ (suporte completo).
- Safari < 17: funcional com degradação aceitável de animações.
- Internet Explorer: não suportado.

---

### RNF-020 — Suportabilidade: Mobile (React Native)

**Origem:** Doc 02 Stacks, seção 4

O app mobile deve suportar as versões de iOS e Android cobertas pelo React Native 0.76.

**Critério de aceite:**
- iOS 15+ e Android 8+ (API Level 26+).
- Testado em dispositivo físico iOS e Android antes de cada release.
- Acessibilidade: VoiceOver (iOS) e TalkBack (Android) testados em fluxos críticos.

---

### RNF-021 — Desempenho de carregamento mobile

**Origem:** `[DECISÃO AUTÔNOMA: experiência mobile adequada para operadores em campo]`

O app mobile deve carregar a tela inicial em até 3 segundos em conexão 4G.

**Critério de aceite:**
- Time to Interactive (TTI) < 3s em conexão 4G simulada.
- Telas principais com skeleton screens. Spinners proibidos.
- Lazy loading de imagens via `expo-image`.

---

### RNF-022 — Internacionalização: Português do Brasil

**Origem:** Doc 01 contexto estratégico

Toda a interface, mensagens de erro, notificações e documentação devem ser em Português do Brasil.

**Critério de aceite:**
- Nenhuma string em inglês visível ao usuário final (exceto nomes próprios de serviços: ZapSign, Celcoin, etc.).
- Erros da API retornados em português.
- Datas no formato `DD/MM/AAAA`. Valores monetários no formato `R$ X.XXX,XX`.

---

### RNF-023 — Monorepo e estrutura de projeto

**Origem:** Doc 02 Stacks, seção 6.1

O projeto deve ser estruturado como monorepo Turborepo + pnpm com a estrutura definida no Doc 02.

**Critério de aceite:**
- Estrutura: `apps/api`, `apps/web`, `apps/mobile`, `packages/shared-types`, `packages/shared-utils`, etc.
- pnpm workspaces configurados. npm e yarn proibidos.
- `turbo.json` com pipelines de build, test e lint.
- `docker-compose.yml` com Redis + RabbitMQ para ambiente local.

---

### RNF-024 — CI/CD: Pipeline automático de qualidade

**Origem:** Doc 02 Stacks, seção 8

Todo código deve passar por pipeline de CI antes de ser deployado.

**Critério de aceite:**
- GitHub Actions: lint → type-check → testes de integração → `prisma migrate deploy` → build → deploy.
- Deploy automatizado apenas após todos os passos passarem.
- Pull requests: CI obrigatório. Merge bloqueado se CI falhar.
- Environment secrets gerenciados via GitHub Secrets (nunca em código).

### 3.2 Definition of Done por RF

Um RF é considerado concluído quando:
1. Implementado no backend (NestJS).
2. Implementado no frontend (React + Vite) e/ou mobile (Expo) quando aplicável.
3. Testes de integração passando no CI.
4. Code review aprovado.
5. Documentação de API atualizada (Swagger).
6. Sem TypeScript errors.
7. Lint e formatter passando.
