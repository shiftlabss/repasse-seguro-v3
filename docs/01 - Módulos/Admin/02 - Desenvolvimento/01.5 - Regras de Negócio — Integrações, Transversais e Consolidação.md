## 2. Módulo de Integração — ZapSign

### RN-121 — Integração ZapSign: Envio de Envelopes

**Contexto:** A integração ZapSign é acionada no módulo de Formalização para coletar assinaturas eletrônicas de Cedente, Cessionário e, quando aplicável, da Construtora.

**Condições de disparo:**
- Operador com perfil ≥ Analista executa ação "Enviar para Assinatura" no módulo Formalização
- O caso está no estado `Em Formalização`
- O documento a ser assinado foi carregado e aprovado na triagem documental

**Fluxo principal (caminho positivo):**
1. Sistema monta o envelope com: documento PDF, lista de signatários (nome, e-mail, CPF), ordem de assinatura definida (Cedente → Cessionário → Construtora, quando aplicável), prazo de expiração do envelope (configurable, default 30 dias corridos)
2. Sistema envia requisição `POST /api/v1/documents` à API ZapSign com autenticação via API Key
3. ZapSign retorna `document_token` e URLs de assinatura individuais por signatário
4. Sistema persiste `zapsign_document_token`, URLs e status `ENVIADO` no registro do caso
5. Sistema dispara notificações por e-mail para cada signatário com link direto de assinatura (via ZapSign)
6. Sistema registra evento na trilha de auditoria: `DOCUMENTO_ENVIADO_ZAPSIGN`, timestamp, operador, token

**Fluxo de erro (caminho negativo):**
- Se a API ZapSign retornar erro 4xx (dados inválidos): sistema exibe mensagem de erro específica ao operador; não registra o envio; operador pode corrigir e reenviar
- Se a API ZapSign retornar erro 5xx ou timeout: sistema registra falha na trilha de auditoria; ativa modo de contingência (RN-036); caso permanece em `Em Formalização`

**Efeitos de estado:** Nenhuma transição de estado do caso — o envio ZapSign não muda o estado; apenas o recebimento de todas as assinaturas habilita o critério

**Mensagem ao operador (erro de rede):** "Não foi possível conectar ao serviço de assinaturas no momento. O caso permanece em Formalização e o envio pode ser tentado novamente."

**Mensagem ao operador (dados inválidos):** "O envelope não pôde ser criado. Verifique se todos os e-mails dos signatários estão corretos e tente novamente."

**Interface de envio ZapSign na Formalização:** [CORRIGIDO: PROBLEMA-043]
- Card "Assinaturas" no painel de Formalização exibe: status de cada signatário (Pendente/Assinado/Recusado/Expirado), barra de progresso (0/3, 1/3, 2/3, 3/3) e botão "Enviar para Assinatura".
- Após envio: cada signatário exibe badge de status individual que atualiza em tempo real via webhook.
- Botão "Reenviar" visível ao lado de signatários com status Pendente (após 5 dias) ou Expirado. Contador de reenvios: "Reenvio X de 3".
- [DECISÃO APLICADA: DEC-012] A barra de progresso de assinaturas usa numeração fracionária (ex: "1/3 assinaturas") em vez de percentual. Justificativa: com apenas 3 signatários, porcentagens (33%, 66%, 100%) são menos intuitivas que contagem direta.

**Banner de contingência ZapSign (RN-123):** [CORRIGIDO: PROBLEMA-044]
- Banner global amarelo no topo do módulo Formalização: "Serviço de assinaturas temporariamente indisponível. Operações de assinatura estão pausadas."
- O banner inclui timestamp do início da indisponibilidade e link para opção de "Registrar assinatura manual" (visível apenas para Coordenador e Master).
- Ao restaurar a integração, o banner desaparece automaticamente e o sistema exibe toast: "Serviço de assinaturas restaurado. Envelopes pendentes sendo reenviados."

**Permissões:**

| Ação | Analista | Coordenador | Gestor Financeiro | Master |
|---|---|---|---|---|
| Enviar envelope | ✅ | ✅ | ✗ | ✅ |
| Reenviar envelope expirado | ✅ | ✅ | ✗ | ✅ |
| Cancelar envelope ativo | ✗ | ✅ | ✗ | ✅ |
| Visualizar status de assinaturas | ✅ | ✅ | ✅ | ✅ |

### RN-122 — Integração ZapSign: Recebimento de Webhooks

**Contexto:** ZapSign notifica o sistema via webhook a cada evento de assinatura (assinado, recusado, expirado).

**Eventos monitorados:**

| Evento ZapSign | Ação no Sistema | Notificação interna |
|---|---|---|
| `document_signed` (um signatário) | Atualiza status individual do signatário no caso | Exibe no painel em tempo real |
| `document_completed` (todos assinaram) | Marca critério "instrumento assinado" como atendido | Notifica Analista responsável |
| `document_refused` | Registra recusa com motivo; caso permanece em `Em Formalização`; alerta Analista | Notificação urgente |
| `document_expired` | Desmarca critério; alerta Analista para reenvio | Notificação urgente |

**Validação do webhook:**
1. Sistema verifica assinatura HMAC-SHA256 do payload usando secret configurado em `Configurações → ZapSign`
2. Se assinatura inválida: descarta payload e registra tentativa na trilha de auditoria (possível ataque)
3. Se assinatura válida: processa evento de forma idempotente (mesmo `document_token` + mesmo evento não gera duplicatas)

**Política de retry (lado do sistema):** O sistema deve responder HTTP 200 em até 5 segundos; caso contrário, ZapSign reprocessará com backoff exponencial (conforme documentação do fornecedor). O sistema deve ser idempotente para reprocessamentos.

**Efeitos de estado:** `document_completed` pode desencadear a avaliação dos 4 critérios de fechamento (RN-023); se todos os 4 critérios forem simultaneamente atendidos, o sistema habilita o botão "Confirmar Fechamento" para o Coordenador

**Mensagem ao Analista (documento recusado):** "O signatário [Nome] recusou a assinatura do documento. Verifique o motivo informado e entre em contato para prosseguir."

**Mensagem ao Analista (documento expirado):** "O prazo de assinatura expirou sem que todos os signatários concluíssem. Reenvie o envelope para reabrir o prazo."

### RN-123 — Integração ZapSign: Contingência (modo offline)

**Contexto:** Referenciado em RN-036 (arquivo 01.2). Detalha os procedimentos quando ZapSign está indisponível.

**Ativação do modo de contingência:**
- Após 2 tentativas consecutivas de envio falhando com timeout (> 30 segundos cada)
- Ou recebimento de status page ZapSign indicando degradação (monitoramento automático)

**Procedimentos em modo de contingência:**
1. Sistema exibe banner global para operadores do módulo Formalização: "Serviço de assinaturas temporariamente indisponível. Operações de assinatura estão pausadas."
2. SLA do caso é automaticamente pausado (RN-061, arquivo 01.3) — o tempo em modo contingência não conta contra o SLA
3. Operadores podem registrar manualmente que as partes assinaram fisicamente, anexando o documento digitalizado como evidência — esta ação requer aprovação de Coordenador
4. Ao restabelecer a integração: sistema tenta re-enviar envelopes pendentes automaticamente; operador é notificado para confirmar se prefere manter o registro manual
5. Registro manual aprovado por Coordenador tem a mesma validade que assinatura eletrônica para fins de critério de fechamento

**[DECISÃO AUTÔNOMA DA-009]:** O prazo máximo em modo de contingência antes de escalar para o Master foi definido como 48 horas. Após 48h sem integração, o Master deve ser notificado por e-mail automático com lista de casos impactados.

**Efeitos de estado:** Nenhum — casos permanecem em `Em Formalização`; SLA pausado não avança

### RN-124 — Integração ZapSign: Gestão de Templates de Documento

**Contexto:** O sistema mantém templates de documentos para geração automática dos instrumentos de cessão.

**Regras de gestão:**
1. Templates são configurados na seção `Configurações → ZapSign` (acesso exclusivo Master — RN-109)
2. Cada template tem versão e data de ativação; alterações são prospectivas (não afetam casos já em andamento)
3. Sistema mantém histórico de todos os templates com versão, data de criação, operador que criou e status (ativo/inativo)
4. O sistema gera o documento final combinando: template ativo + dados do caso no momento da geração
5. Uma vez gerado e enviado para assinatura, o documento é imutável — alterações no template não afetam envelopes já enviados
6. O sistema persiste o PDF gerado associado ao caso (não apenas o template)

**[DECISÃO AUTÔNOMA DA-010]:** O MVP não implementa geração automática de documentos; o sistema recebe o PDF já preparado pelo operador. A integração com geração automática a partir de template é roadmap pós-MVP.

## 3. Módulo de Integração — Parceiro Financeiro (Conta Escrow)

### RN-125 — Integração Escrow: Abertura de Conta

**Contexto:** Cada caso qualificado requer uma Conta Escrow individual (RN-020.a, arquivo 01.2). A abertura é feita via integração com o parceiro financeiro.

**[DEFINIÇÃO PENDENTE DP-005]:** O parceiro financeiro para a Conta Escrow não foi definido. Duas opções estão em análise:
- **Opção A:** Banco parceiro com API proprietária (maior controle, menor custo por transação, integração customizada necessária)
- **Opção B:** Plataforma de escrow como serviço (integração padronizada, maior custo por transação, menor esforço de desenvolvimento)

**Fluxo de abertura (contrato esperado, independente do parceiro):**
1. Sistema envia requisição de abertura com: identificador único do caso (UUID), valor esperado do depósito, dados do Cedente (nome, CPF/CNPJ), dados do Cessionário (nome, CPF/CNPJ), finalidade declarada ("Cessão Imobiliária — Repasse Seguro")
2. Parceiro retorna: número da conta escrow, agência, dados bancários para depósito, identificador único da conta no sistema do parceiro
3. Sistema persiste dados da conta escrow associados ao caso; exibe dados para o Coordenador no módulo Formalização
4. Sistema registra evento `CONTA_ESCROW_ABERTA` na trilha de auditoria

**Condição de disparo:** Analista move caso para estado `Em Formalização` → sistema abre conta escrow automaticamente (sem ação manual do operador)

**Efeitos de estado:** Nenhuma transição de estado do caso; a conta escrow é um pré-requisito que passa a estar disponível para o Cessionário

**Mensagem ao Coordenador (conta aberta):** "A conta escrow foi criada para este caso. Os dados bancários para depósito foram disponibilizados na aba Formalização."

**Mensagem ao Coordenador (falha na abertura):** "Não foi possível criar a conta escrow automaticamente. Entre em contato com o suporte para abertura manual."

### RN-126 — Integração Escrow: Confirmação de Depósito

**Contexto:** O parceiro financeiro notifica o sistema quando um depósito é identificado na conta escrow.

**Fluxo de confirmação:**
1. Parceiro envia webhook com: número da conta, valor depositado, data/hora do depósito, identificador da transação no parceiro
2. Sistema valida assinatura do webhook (HMAC ou certificado mútuo TLS, conforme parceiro)
3. Sistema compara valor depositado com valor esperado (com tolerância de R$0,50 — RN-020.b)
4. Se valor dentro da tolerância: marca critério "depósito na Conta Escrow" como atendido; dispara avaliação dos 4 critérios (RN-023); notifica Analista e Coordenador
5. Se valor abaixo da tolerância: registra depósito parcial; não marca critério; notifica Coordenador para contatar Cessionário (RN-035)
6. Se valor acima da tolerância: aceita o depósito; registra excedente; notifica Gestor Financeiro para definir tratamento do excedente

**Política de reconciliação manual:** Caso o webhook não chegue em 24 horas após o prazo de depósito, o Gestor Financeiro pode registrar manualmente a confirmação do depósito mediante upload de comprovante bancário, sujeito a aprovação do Master

**Mensagem ao Cessionário (depósito parcial):** "Identificamos um depósito inferior ao valor acordado. Por favor, realize o complemento para que o processo possa continuar."

**Mensagem ao Coordenador (depósito confirmado):** "O depósito foi confirmado para o caso [ID]. O critério escrow está atendido."

### RN-127 — Integração Escrow: Distribuição e Estorno

**Contexto:** Após o Fechamento (RN-024), o sistema solicita ao parceiro a distribuição dos valores da conta escrow.

**Fluxo de distribuição:**
1. Sistema calcula a distribuição (Cedente: valor de repasse; Repasse Seguro: comissão 20%; encargos bancários deduzidos — RN-022)
2. Sistema aguarda janela de 24 horas de notificação prévia (RN-021)
3. Após janela: sistema envia requisição de distribuição ao parceiro com: conta destino Cedente (dados bancários já cadastrados), valor Cedente, conta destino Repasse Seguro, valor comissão, referência do caso
4. Parceiro executa TED/PIX e retorna confirmação com comprovante de cada transferência
5. Sistema persiste comprovantes; atualiza status da Conta Escrow para `Distribuído`; registra evento na auditoria; notifica Cedente e Gestor Financeiro
6. Caso passa automaticamente para estado `Concluído`

**Fluxo de estorno (RN-089):**
1. Gestor Financeiro solicita estorno total via módulo Financeiro
2. Sistema verifica que não houve distribuição prévia; bloqueia estorno parcial
3. Sistema envia requisição de devolução ao parceiro com identificação da conta e valor integral
4. Parceiro processa devolução para a conta de origem do depósito (Cessionário)
5. Sistema atualiza status Conta Escrow para `Estornado`; registra evento; notifica partes

**[DECISÃO AUTÔNOMA DA-011]:** Em caso de falha na transferência bancária (conta destino incorreta, banco indisponível), o sistema tentará 3 vezes com intervalo de 1 hora. Após 3 falhas, o Gestor Financeiro é alertado para intervenção manual com o parceiro.

### RN-128 — Integração Escrow: Retenção e Período de Custódia

**Contexto:** Após o Fechamento e antes da distribuição, os valores ficam retidos na Conta Escrow por 15 dias corridos (período de custódia).

**Regras do período de custódia:**
1. Período de 15 dias inicia-se a partir da confirmação do Fechamento (transição de estado — não da data do depósito)
2. Durante o período de custódia: sistema bloqueia requisições de distribuição automática no parceiro; apenas o Master pode autorizar distribuição antecipada
3. O período de custódia serve como proteção para reversão consensual (RN-037) — se reversão for solicitada dentro de 15 dias, a Conta Escrow ainda está retida e o estorno é direto
4. Após 15 dias: sistema agenda automaticamente a distribuição (aviso 24h → distribuição)
5. O parceiro financeiro deve suportar instrução de "retenção programada com liberação automática em data específica" ou o sistema deve gerenciar isso via cron job interno

**Efeitos de estado da Conta Escrow:**

```
stateDiagram-v2
    [*] --> Aguardando_Abertura
    Aguardando_Abertura --> Aberta : parceiro confirma abertura
    Aberta --> Aguardando_Deposito : dados enviados ao Cessionário
    Aguardando_Deposito --> Depositado : webhook confirmação (dentro tolerância)
    Aguardando_Deposito --> Deposito_Parcial : valor abaixo da tolerância
    Deposito_Parcial --> Depositado : complemento recebido
    Depositado --> Em_Retencao : caso atinge Fechamento
    Em_Retencao --> Aguardando_Distribuicao : 15 dias corridos decorridos
    Aguardando_Distribuicao --> Distribuido : transferências confirmadas
    Em_Retencao --> Estornado : reversão aprovada dentro de 15 dias
    Aguardando_Distribuicao --> Estornado : Master autoriza estorno
    Distribuido --> [*]
    Estornado --> [*]
```

## 4. Regras Transversais

As regras desta seção aplicam-se a **todos os módulos** sem exceção. Elas não foram duplicadas nos arquivos anteriores para evitar conflito; qualquer módulo que precise referenciar estas regras deve citar o número da RN desta seção.

### RN-129 — Audit Trail Imutável (Consolidação)

**Contexto:** Reúne e detalha a regra de auditoria mencionada em RN-016 (01.1), RN-105 (01.4) e em múltiplas RNs de integração.

**Definição:** Todo evento que altere estado, dados, configuração ou acesso deve ser registrado de forma imutável. "Imutável" significa que registros de auditoria não podem ser modificados, deletados ou sobrescritos por nenhum perfil de operador, incluindo o Master.

**Estrutura obrigatória de cada registro de auditoria:**

| Campo | Tipo | Obrigatório | Descrição |
|---|---|---|---|
| `id` | UUID v4 | Sim | Identificador único do evento |
| `timestamp` | ISO 8601 com fuso horário | Sim | Momento exato do evento (UTC) |
| `event_type` | Enum | Sim | Tipo do evento (ver tabela abaixo) |
| `actor_id` | UUID | Sim | ID do operador ou sistema que causou o evento |
| `actor_profile` | Enum | Sim | Perfil do ator no momento do evento |
| `case_id` | UUID | Condicional | ID do caso, quando aplicável |
| `entity_type` | String | Sim | Entidade afetada (Case, User, Configuration, etc.) |
| `entity_id` | UUID | Sim | ID da entidade afetada |
| `previous_value` | JSON | Condicional | Estado anterior (para alterações de dados) |
| `new_value` | JSON | Condicional | Estado novo (para alterações de dados) |
| `ip_address` | String | Sim | IP de origem da requisição |
| `user_agent` | String | Sim | User agent do cliente |
| `correlation_id` | UUID | Sim | ID da requisição HTTP (para rastreamento end-to-end) |

**Categorias de eventos obrigatoriamente auditados:**

| Categoria | Exemplos |
|---|---|
| Autenticação | Login, logout, falha de login, 2FA, sessão expirada |
| Autorização | Tentativa de acesso negada por RBAC |
| Ciclo de vida do caso | Toda transição de estado |
| Dados do caso | Alteração em qualquer campo do caso |
| Propostas | Criação, aceite, recusa de proposta |
| Assinaturas | Envio/recebimento de envelope ZapSign |
| Escrow | Abertura de conta, depósito, distribuição, estorno |
| Usuários | Criação, alteração, suspensão, reativação |
| Configurações | Toda alteração de parâmetro |
| Agentes IA | Takeover, release, escalação, calibração |
| Exportações | Toda exportação de dados |
| LGPD | Solicitações de exclusão, anonimização |

**Acesso ao audit trail:** Somente via módulo Relatórios → Trilha de Auditoria (RN-105), com filtros obrigatórios para evitar dump completo. Export em CSV/JSON permitido para Master.

**Retenção:** Registros de auditoria são retidos por **5 anos** (prazo legal para contratos imobiliários no Brasil). Após 5 anos, podem ser arquivados em storage frio, mas não deletados.

**[DECISÃO AUTÔNOMA DA-012]:** O storage de auditoria deve ser separado do banco de dados principal (tabela append-only em schema dedicado ou serviço separado) para garantir que falhas no banco principal não afetem a integridade da auditoria.

### RN-130 — Lock Otimista para Edição Concorrente

**Contexto:** Múltiplos operadores podem visualizar e editar o mesmo caso simultaneamente. O lock otimista previne sobrescrita silenciosa de dados.

**Mecanismo:**
1. Toda entidade editável carrega um campo `version` (integer, auto-incrementado a cada `UPDATE`)
2. Quando operador abre um formulário de edição: frontend registra a versão atual da entidade
3. Ao submeter: requisição inclui `{ ..dados_novos, version: N }` no payload
4. Backend verifica: se `entidade.version == N` (sem alteração concorrente) → aplica mudança e incrementa version; se `entidade.version != N` (alguém alterou enquanto operador editava) → retorna HTTP 409 Conflict
5. Frontend exibe modal de conflito com as duas versões lado a lado (versão do operador vs. versão atual do banco)
6. Operador escolhe: "Sobrescrever com meus dados" (requer perfil ≥ Coordenador para campos críticos) ou "Descartar e ver versão atual"

**Campos que nunca podem ser sobrescritos unilateralmente (requerem aprovação do Master):**
- Status/estado do caso
- Comissão calculada
- Dados bancários do Cedente
- Critérios de Fechamento já marcados como atendidos

**Mensagem ao operador (conflito detectado):** "Este registro foi modificado por outro operador enquanto você editava. Revise as alterações antes de confirmar."

**Modal de conflito de edição concorrente:** [CORRIGIDO: PROBLEMA-045]
- O modal de conflito (HTTP 409) exibe comparação lado a lado: "Seus dados" (esquerda) vs. "Dados atuais" (direita).
- Campos com diferença são destacados em amarelo.
- Dois botões: "Sobrescrever com meus dados" (disponível para Coordenador+ em campos não críticos) e "Descartar e ver versão atual" (disponível para todos).
- Para campos críticos (status, comissão, dados bancários): apenas "Descartar e ver versão atual" está disponível. O operador precisa solicitar ao Master para sobrescrever.
- Após resolução: toast informando o resultado escolhido.

**Proteção contra duplo submit (RN-133):** [CORRIGIDO: PROBLEMA-046]
- Ao clicar em qualquer botão de ação destrutiva/mutativa: botão é imediatamente desabilitado e exibe spinner.
- Se a requisição demora mais de 5 segundos: exibir texto "Processando, por favor aguarde..." abaixo do botão.
- Se timeout (30 segundos): reabilitar botão + toast de erro "A operação demorou mais que o esperado. Tente novamente."
- [DECISÃO APLICADA: DEC-013] O timeout de feedback ao usuário é 30 segundos. Justificativa: operações que envolvem integrações externas (ZapSign, parceiro escrow) podem levar mais tempo; 30 segundos é o limite antes de o usuário assumir que algo falhou.

### RN-131 — Isolamento de Dados por Perfil (Consolidação)

**Contexto:** Consolida RN-015 (01.1), RN-069 (01.3) e referências em módulos de Negociação e Triagem.

**Princípio:** Um operador só acessa dados que seu perfil autoriza **E** que pertencem ao escopo de sua visibilidade. Perfis mais altos têm escopo mais amplo, não acesso irrestrito a dados pessoais.

**Matriz de isolamento de dados sensíveis:**

| Dado Sensível | Analista | Coordenador | Gestor Financeiro | Master |
|---|---|---|---|---|
| CPF/CNPJ do Cedente (completo) | Mascarado (***.***.***-**) | Completo | Completo | Completo |
| CPF/CNPJ do Cessionário (completo) | Mascarado | Completo | Completo | Completo |
| Dados bancários do Cedente | Oculto | Visível | Visível | Visível |
| Valor financeiro da negociação | Visível | Visível | Visível | Visível |
| Identidade do Cessionário em fase de negociação | Anonimizado | Visível | ✗ (não acessa negociação) | Visível |
| Logs de auditoria de outros usuários | ✗ | Somente casos sob sua coordenação | Somente eventos financeiros | Completo |
| Dados de outros Cedentes (casos distintos) | ✗ | ✗ | ✗ | ✅ |

**Regra de anonimização em Negociação:** Durante as fases de proposta e contraproposta, o Analista visualiza o Cessionário como "Cessionário Interessado #N" sem nome, CPF ou contato. A identidade é revelada somente após aceite formal de proposta ou quando o Coordenador deliberadamente a revela (com registro em auditoria).

### RN-133 — Proteção Contra Duplo Submit

**Contexto:** Ações que geram mutações de estado (transições de caso, criação de proposta, envio de envelope, distribuição escrow) devem ser protegidas contra submissão duplicada.

**Implementação obrigatória:**
1. Ao disparar ação: botão é imediatamente desabilitado e exibe estado de carregamento (spinner)
2. Se a requisição retornar sucesso (2xx): botão permanece desabilitado até que a UI seja recarregada com o novo estado

### RN-134 — Sessão e Timeout de Autenticação

**Contexto:** Consolida USR-08 (01.3) e referências de auth em RN-005 (01.1).

**Regras:**
1. Sessão ativa por inatividade conforme o perfil do usuário (CORREÇÃO #02):
   - **Operadores Admin** (Analista, Coordenador, Gestor Financeiro, Master): **8 horas** de inatividade.
   - **Cedentes:** **24 horas** de inatividade.
   - **Cessionários:** **30 minutos** de inatividade (conforme módulo Cessionário/01.1 RN-004).
   O timer é resetado a cada requisição autenticada.
2. Ao atingir 7h45min sem atividade: sistema exibe modal de aviso não bloqueante "Sua sessão expirará em 15 minutos. Clique aqui para continuar."
3. Ao atingir 8h de inatividade: sessão encerrada; tokens invalidados no Redis (blacklist); usuário redirecionado para login com mensagem: "Sua sessão expirou por inatividade. Faça login novamente."
4. O encerramento de sessão não descarta dados preenchidos no formulário ativo — ao logar novamente, o frontend deve tentar restaurar o rascunho via localStorage (com TTL de 4 horas)
5. JWT de acesso com expiração de 15 minutos; refresh token com expiração de 8 horas; refresh token de uso único (rotação a cada uso)
6. Login em novo dispositivo/IP diferente: notificação por e-mail ao usuário (não bloqueia — apenas alerta)

**[DECISÃO AUTÔNOMA DA-014]:** A sessão de 8 horas se aplica ao token de refresh. O token de acesso (15min) é renovado automaticamente pelo frontend antes de expirar, sem interação do usuário, desde que o refresh token ainda seja válido.

### RN-136 — Tratamento de Erros de Rede e Feedback ao Usuário

**Contexto:** O sistema deve ter comportamento definido para todos os cenários de falha de rede e erros de API.

**Categorias e respostas:**

| Tipo de Erro | HTTP Status | Comportamento Frontend | Mensagem ao Usuário |
|---|---|---|---|
| Sem conexão / offline | — (timeout) | Banner persistente no topo; operações de leitura usam cache; operações de mutação são bloqueadas | "Você está offline. Verifique sua conexão para continuar." |
| Erro de validação | 400 | Exibir erros inline nos campos correspondentes; não exibir toast | Mensagem específica por campo |
| Não autorizado (sessão expirada) | 401 | Redirecionar para login preservando URL atual | "Sua sessão expirou. Faça login para continuar." |
| Acesso negado (perfil insuficiente) | 403 | Exibir modal de acesso negado; não redirecionar | "Você não tem permissão para realizar esta ação." |
| Recurso não encontrado | 404 | Exibir página de erro 404 com link para voltar | "Esta página não foi encontrada." |
| Conflito (lock otimista) | 409 | Exibir modal de conflito com opções (RN-130) | "Este registro foi modificado por outro operador." |
| Erro interno do servidor | 500 | Exibir toast de erro com option de tentar novamente; registrar no Sentry/observabilidade | "Ocorreu um erro inesperado. Tente novamente ou contate o suporte." |
| Rate limit atingido | 429 | Desabilitar ação por N segundos (conforme Retry-After header) | "Muitas tentativas. Aguarde [N] segundos antes de tentar novamente." |

**Regra adicional:** Nenhuma mensagem de erro deve expor detalhes técnicos (stack trace, nomes de tabela, queries SQL) ao usuário final. Esses detalhes devem estar disponíveis apenas nos logs de observabilidade.

**Hierarquia de feedback de erros:** [CORRIGIDO: PROBLEMA-047]
- Erros de validação (400): feedback inline nos campos do formulário. Sem toast. O foco do teclado é movido para o primeiro campo com erro.
- Erros de autenticação (401): redirecionamento para login com preservação da URL de retorno. Toast informativo.
- Erros de permissão (403): modal overlay com mensagem e botão "Voltar à Dashboard". Sem redirecionamento automático.
- Erros internos (500): toast com botão "Tentar novamente". Se o erro persistir em 3 tentativas, exibir: "Estamos com problemas técnicos. Se o erro persistir, contate o suporte."
- Modo offline: banner fixo amarelo no topo (acima da sidebar) que persiste até a reconexão. Botões de ação ficam desabilitados com tooltip "Indisponível offline."

**Copy de mensagens de erro — tom e estrutura:** [CORRIGIDO: PROBLEMA-048]
- Todas as mensagens seguem formato: [O que aconteceu] + [O que fazer]. Exemplo: "Sua sessão expirou por inatividade. Faça login novamente."
- Tom: neutro e empático. Não usar: "Erro!", "Ops!", exclamações excessivas ou linguagem técnica.
- Máximo 2 frases por mensagem de erro.
- [DECISÃO APLICADA: DEC-014] Mensagens de erro não usam termos técnicos como "HTTP 500", "timeout" ou "servidor". Justificativa: operadores do Admin são profissionais de negócio, não engenheiros; termos técnicos geram confusão e ansiedade desnecessária.

### RN-138 — Webhook: Política de Envio e Retry

**Contexto:** O sistema envia webhooks para sistemas externos configurados por integradores (funcionalidade de plataforma/API). Consolida referências de retry em módulos de notificação.

**Política de envio de webhooks de saída:**
1. Payload enviado como `POST` com `Content-Type: application/json`
2. Autenticação via `X-Repasse-Seguro-Signature` (HMAC-SHA256 do payload com secret do integrador)
3. Timeout de resposta: 30 segundos
4. **Política de retry:** 3 tentativas no total, com intervalos progressivos:
   - Tentativa 1: imediata
   - Tentativa 2: 30 segundos após falha
   - Tentativa 3: 5 minutos após segunda falha
   - Tentativa 4 (final): 30 minutos após terceira falha
5. Após 4 tentativas falhando: webhook marcado como `FALHOU`; alerta enviado ao Master; event registrado na auditoria

**Eventos publicáveis via webhook:**
- Toda transição de estado do caso
- Confirmação de depósito escrow
- Conclusão de todas as assinaturas (ZapSign)
- Fechamento do caso
- Cancelamento do caso

**[DECISÃO AUTÔNOMA DA-015]:** No MVP, webhooks de saída para integradores externos não estão disponíveis. Esta funcionalidade é roadmap pós-MVP para cases de integração com ERPs imobiliários e CRMs de parceiros.

### RN-150 — Auditoria Financeira Imutável

> Origem: FIN-07 (arquivo de entrada)

**Contexto:** Toda movimentação financeira na plataforma (depósito, distribuição, estorno, congelamento) deve gerar um registro de auditoria imutável. Consolida a regra FIN-07 do arquivo de entrada com RN-129 (Audit Trail Geral).

**Fluxo principal:**
1. Qualquer movimentação financeira ocorre em uma Conta Escrow: depósito confirmado pelo parceiro, distribuição automática ou manual, congelamento, desbloqueio ou estorno.
2. O sistema registra automaticamente um evento de auditoria financeira contendo: tipo da movimentação, valor, data e hora, responsável pela ação (operador ou sistema), Conta Escrow de origem, status anterior e novo status.
3. **O registro é criado automaticamente pelo sistema** — nenhum operador precisa acioná-lo manualmente.
4. **O Gestor Financeiro não pode editar registros passados** de auditoria financeira. Nenhum perfil pode alterar ou excluir esses registros, nem mesmo o Master.
5. **Se o Gestor Financeiro tentar alterar ou excluir um registro:** a operação é bloqueada e registrado o acesso não autorizado na trilha de auditoria de segurança.
6. **Consequência se violada:** Movimentações sem registro imutável impossibilitam auditoria externa, impugnação de disputas financeiras e conformidade com regulação do parceiro financeiro da Conta Escrow.

### RN-151 — Atualização em Tempo Real do Pipeline (até 5 segundos)

> Origem: P-03 (arquivo de entrada)

**Contexto:** O Pipeline é a tela de trabalho colaborativo central. Quando um operador altera o status de um caso, todos os demais operadores logados devem ver a atualização em tempo próximo ao real.

**Fluxo principal:**
1. Um operador altera o status de um caso a partir de qualquer tela da plataforma (Pipeline, Triagem, Negociação, Formalização).
2. O sistema propaga a atualização de status para todos os operadores que estão com o Pipeline aberto.
3. **O Pipeline de todos os operadores logados atualiza automaticamente em até 5 segundos** após a mudança, via WebSocket (quando disponível) ou polling com intervalo de 5 segundos.
4. **Se o WebSocket não estiver disponível:** o sistema usa polling com intervalo de 5 segundos como fallback automático, sem necessidade de ação do operador.
5. **Se a atualização em tempo real falhar:** o operador pode forçar a sincronização clicando no botão "Atualizar" no Pipeline. O sistema exibe badge discreto "Última atualização: X minutos atrás" para indicar que os dados podem estar desatualizados.
6. **Consequência se violada:** Operadores trabalhando com versões desatualizadas do Pipeline podem tomar ações conflitantes sobre o mesmo caso, resultando em erros de concorrência e perda de trabalho.

### RN-152 — DisputaFormal: prazo, responsáveis e saída

**Contexto:** Estado ativado quando a mediação interna falha após 10 dias úteis sem acordo (conforme RN-038). É o estado de maior risco operacional e financeiro da plataforma.

**Responsáveis:**
- **Master:** responsável interno pela condução e acompanhamento da disputa.
- **Externo:** parceiro jurídico previamente contratado pelo Repasse Seguro para mediação extrajudicial. O Master aciona o parceiro externo via contato direto no prazo de 2 dias úteis após a ativação do estado.

**Prazo máximo em DisputaFormal:** 90 dias corridos a partir da data de ativação.
- A cada 30 dias: o sistema gera alerta automático ao Master com resumo do tempo decorrido e status da resolução.
- No 90º dia sem resolução: o sistema notifica o Master que o caso deve ser encerrado por via judicial. O Escrow permanece retido até ordem judicial ou acordo formal documentado.

**Gatilhos de saída do estado DisputaFormal:**

| Desfecho | Quem confirma | Ação no sistema |
|---|---|---|
| Acordo extrajudicial com manutenção do negócio | Master (anexa documento assinado) | Caso volta para Pós Fechamento; Escrow retoma contagem de 15 dias |
| Acordo extrajudicial com cancelamento | Master (anexa documento assinado) | Caso vai para Cancelado; Escrow estorna valor integral ao Cessionário |
| Ordem judicial de liberação ao Cedente | Master (anexa decisão judicial) | Caso vai para Concluído; Escrow distribui conforme decisão |
| Ordem judicial de devolução ao Cessionário | Master (anexa decisão judicial) | Caso vai para Cancelado; Escrow estorna conforme decisão |

**Regras operacionais:**
1. Nenhuma movimentação financeira na Conta Escrow é permitida durante DisputaFormal sem documento formal anexado ao dossiê.
2. O Analista e o Coordenador ficam com acesso somente leitura ao caso. Apenas Master pode executar ações.
3. O prazo de 90 dias não é prorrogável sem decisão explícita do Master com justificativa registrada no dossiê.

**Mensagem ao Master ao ativar DisputaFormal:**
"O caso [ID] entrou em Disputa Formal. Você tem 2 dias úteis para acionar o parceiro jurídico externo. Os valores permanecerão retidos na Conta Escrow até resolução formal."

**Consequência se violada:** Sem prazo e sem processo definido, o caso pode ficar retido indefinidamente, bloqueando capital do Cessionário e gerando risco jurídico para o Repasse Seguro.

## 11. Casos de Borda Transversais

| ID | Cenário | Módulos envolvidos | Comportamento esperado |
|---|---|---|---|
| CB-T01 | Operador perde conexão durante transição de estado | Todos | Lock otimista detecta versão desatualizada no retry; sistema pergunta se quer reprocessar |
| CB-T02 | Dois Coordenadores tentam fechar o mesmo caso simultaneamente | Formalização | Apenas o primeiro bem-sucedido; segundo recebe HTTP 409; modal de conflito |
| CB-T03 | ZapSign e escrow ficam indisponíveis ao mesmo tempo | Formalização | SLA pausado para ambos; banner duplo; Master notificado; prazo máximo 48h |
| CB-T04 | Webhook de assinatura chega após cancelamento do caso | Formalização, Core | Webhook aceito com HTTP 200 (idempotência); evento registrado na auditoria; nenhuma ação adicional |
| CB-T05 | Solicitação LGPD para exclusão de caso ainda ativo | LGPD, Core | Sistema informa que exclusão não é possível enquanto há obrigação contratual ativa; dados anonimizados somente após conclusão ou cancelamento |
| CB-T06 | Master tenta excluir o único registro de auditoria | Auditoria | Operação bloqueada; sem opção de exclusão na UI; qualquer tentativa via API retorna 403 |
| CB-T07 | Token de acesso expira durante upload de documento grande | Formalização | Silent refresh bem-sucedido não interrompe upload; se refresh falhar, upload é cancelado com mensagem e opção de re-login sem perder o arquivo (salvo em localStorage temporário) |
| CB-T08 | Operador com 2 abas abre o mesmo caso para edição | Todos | Cada aba tem sua versão do lock otimista; a segunda submissão receberá 409 |
