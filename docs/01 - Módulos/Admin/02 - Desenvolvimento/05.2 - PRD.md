## 1. Módulo: Comissões e Conta Escrow

### RF-001 — Cálculo automático de comissão do Cedente por cenário

**Origem:** `Conforme RN-018 (Parte 01.2)`

O sistema deve calcular automaticamente a comissão do Cedente ao atingir o status Fechamento, com base no cenário de retorno registrado no caso.

**Critério de aceite:**
- Cenário A: sistema calcula comissão = R$ 0,00 e registra automaticamente. QA valida que nenhuma comissão é debitada do Cedente.
- Cenários B, C, D: sistema calcula 20% × (Valor Recuperado − Valor Distrato Referência). QA valida com os exemplos: Cenário B (pagou R$300k, recupera R$300k → comissão R$30k), Cenário D (pagou R$500k, recupera R$750k → comissão R$100k).
- Fatura de comissão criada com status "Pendente" imediatamente após cálculo.
- Percentual de comissão configurável entre 5% e 50% pelo Master (RN-120). Alterações afetam apenas novos casos.

---

### RF-002 — Cálculo automático de comissão do Cessionário sobre o Delta

**Origem:** `Conforme RN-019 (Parte 01.2)`

O sistema deve calcular a comissão do Cessionário como 20% do Delta (Tabela Atual − Tabela do Contrato).

**Critério de aceite:**
- Delta positivo: comissão = 20% × Δ. QA valida: Tabela Atual R$700k, Contrato R$500k → Delta R$200k → Comissão R$40k.
- Delta zero ou negativo: comissão variável = R$0. Sistema verifica configuração de exceção Cenário A.
- Exceção Cenário A com Δ=0 ativa: comissão = 20% × Valor Pago pelo Cedente.
- Hierarquia de fontes para Tabela Atual obrigada: 1ª Construtora, 2ª Oferta pública, 3ª Laudo. Sem fonte documentada no dossiê → caso não pode ser fechado.
- QA testa Delta negativo: comissão = R$0 e caso prossegue sem bloqueio adicional.

---

### RF-003 — Aprovação com 4 olhos para Tabela Atual em casos de alto valor

**Origem:** `Conforme RN-019.a (Parte 01.2)`

O sistema deve exigir aprovação do Coordenador quando o Delta calculado for igual ou superior a R$100.000.

**Critério de aceite:**
- Delta ≥ R$100.000: sistema bloqueia critério de Fechamento e envia notificação ao Coordenador com resumo do Delta e fonte usada.
- Delta < R$100.000: critério habilitado sem necessidade de aprovação adicional.
- Coordenador confirma: carimbo duplo registrado (Analista + Coordenador) com timestamp de ambos.
- Coordenador contesta: Analista notificado com motivo. Caso permanece em Formalização.
- Limiar de R$100.000 configurável pelo Master em Configurações → Comissões → "Limiar de revisão do Delta".

---

### RF-004 — Criação automática de Conta Escrow individual por caso

**Origem:** `Conforme RN-020, RN-020.a (Parte 01.2)`

O sistema deve criar automaticamente uma Conta Escrow individual via Celcoin ao caso mudar para status Em Formalização.

**Critério de aceite:**
- Ao mudar status do caso para Em Formalização: sistema dispara criação de Conta Escrow via API Celcoin em até 30 segundos.
- Conta criada com identificador único vinculado ao caso. Dois casos nunca compartilham a mesma conta.
- Status inicial da Conta: "Aberta". Visível no painel de Formalização do Analista.
- Em caso de falha na API Celcoin: sistema registra falha no audit trail, exibe banner de contingência e notifica o Analista para ação manual (conforme RN-128).

---

### RF-005 — Depósito integral na Conta Escrow com tolerância de R$0,50 e responsabilidade de taxas

**Origem:** `Conforme RN-020.b (Parte 01.2)`, `Conforme RN-049 (Parte 01.2)`, `Conforme RN-022 (Parte 01.2)` [CORRIGIDO: PROBLEMA-001] [DECISÃO APLICADA: DEC-001]

O sistema deve aceitar depósito como confirmado apenas quando o valor recebido for igual ao esperado com tolerância máxima de ±R$0,50, e as instruções de depósito devem informar claramente a responsabilidade sobre taxas bancárias.

**Critério de aceite:**
- Depósito dentro da tolerância (±R$0,50): status muda de "Aberta" para "Depósito Confirmado". Analista notificado em tempo real.
- Depósito parcial (abaixo da margem): status permanece "Aguardando depósito". Cessionário notificado para complementar o valor.
- Depósito em moeda estrangeira ou conta internacional: recusado pelo parceiro Celcoin, status não muda.
- Excedente de até R$0,50: registrado como "Crédito residual" para devolução na distribuição.
- **Responsabilidade sobre taxas (RN-022):** as instruções de depósito enviadas ao Cessionário devem incluir aviso explícito: "As taxas de transferência (TED, PIX, DOC) são de responsabilidade do Cessionário e devem ser somadas ao valor a depositar." As taxas de distribuição da Conta Escrow (Celcoin → Cedente, Celcoin → RS) são absorvidas pelo parceiro financeiro. Se o parceiro repassar taxas sobre a distribuição, o valor é descontado da comissão do RS, nunca do valor do Cedente.
- QA: verificar que instrução de depósito exibe aviso de responsabilidade de taxa. Testar cenário de depósito de R$0,30 a menos (não confirma) e R$0,30 a mais (confirma com crédito residual).

---

### RF-006 — Modalidade de depósito fracionado para Cessionários Qualificados

**Origem:** `Conforme RN-020.c (Parte 01.2)`

O sistema deve permitir depósito em 2 etapas (50%+50%) para Cessionários com perfil "Qualificado" habilitado pelo Master.

**Critério de aceite:**
- Master ativa modalidade Cessionário Qualificado: campo disponível nas configurações de perfil do Cessionário.
- 1º depósito (50%): status muda para "Parcialmente Depositado". Assinaturas e anuência podem ser iniciadas.
- 2º depósito: sistema notifica Cessionário com 5 dias úteis de antecedência. Prazo configurável pelo Master.
- Fechamento habilitado apenas com 100% confirmado — depósito fracionado não altera os 4 critérios de RN-023.
- Não-pagamento do 2º depósito no prazo: Analista notificado, fluxo de inadimplência iniciado. Sinal de 50% retido.
- Cessionários não qualificados: modalidade indisponível, fluxo integral de RN-020.

---

### RF-007 — Distribuição automática no 15º dia pós-Fechamento

**Origem:** `Conforme RN-021 (Parte 01.2)`

O sistema deve distribuir automaticamente os valores da Conta Escrow no 15º dia após Fechamento sem reversão.

**Critério de aceite:**
- Sistema verifica diariamente casos em Pós-Fechamento. No 14º dia: Gestor Financeiro notificado com resumo dos valores.
- No 15º dia sem bloqueio: distribuição automática executada. Cedente recebe (Valor Recuperado − Comissão Cedente). RS recebe (Comissão Cedente + Comissão Cessionário). Cessionário recebe crédito residual se houver.
- Bloqueio pelo Gestor Financeiro: justificativa obrigatória (mín. 20 caracteres). Solicitação enviada ao Master para aprovação em até 24h.
- Master não responde em 24h: distribuição retomada automaticamente.
- Conta Escrow muda de "Em Distribuição" para "Distribuída" após execução.

---

### RF-008 — Estorno integral obrigatório (sem estorno parcial)

**Origem:** `Conforme RN-141 (Parte 01.2)`, `Conforme RN-054 (Parte 01.2)`

O sistema deve impedir qualquer estorno parcial e garantir que todo estorno seja do valor integral depositado.

**Critério de aceite:**
- Tentativa de estorno com valor inferior ao total depositado: sistema bloqueia e exibe mensagem "O estorno deve ser do valor integral depositado."
- Reversão confirmada: valor integral depositado pelo Cessionário é devolvido em até 3 dias úteis (prazo bancário) `[DECISÃO AUTÔNOMA: 3 dias úteis alinhado com SLA bancário padrão de TED/PIX]`.
- Conta Escrow muda de "Congelada" para "Estornada" após confirmação.
- QA: testar API de estorno com valor parcial — deve retornar erro 422 com mensagem padronizada.

---

### RF-009 — Distribuição correta no Cenário A (comissão Cedente zerada, comissão Cessionário mantida)

**Origem:** `Conforme RN-142 (Parte 01.2)`

No Cenário A, o sistema deve distribuir sem comissão do Cedente mas manter a comissão do Cessionário para o RS.

**Critério de aceite:**
- Cenário A: preview da distribuição mostra comissão Cedente = R$0,00. Cedente recebe Valor Recuperado integral.
- Comissão do Cessionário (20% sobre valor que Cessionário pagou acima do saldo devedor) vai para RS.
- Gestor Financeiro deve confirmar breakdown antes da execução.
- QA: testar que Cenário A não zera comissão do Cessionário indevidamente.

## 2. Módulo: Fechamento e Validações

### RF-010 — Verificação e exibição dos 4 critérios de Fechamento

**Origem:** `Conforme RN-023 (Parte 01.2)`, `Conforme RN-032 (Parte 01.2)`

O sistema deve exibir painel com os 4 critérios de Fechamento e verificar seu status em tempo real.

**Critério de aceite:**
- Painel exibe 4 cards em grid 2×2: Instrumento assinado, Preço confirmado por evidência, Anuência da construtora, Depósito confirmado.
- Cada card exibe: status (⏳/✅), data de cumprimento, link para evidência.
- Status atualizado automaticamente ao receber webhook ZapSign (RN-122) ou confirmação Celcoin (RN-126).
- Com qualquer critério pendente: botão "Confirmar Fechamento" desabilitado com tooltip listando pendências.
- Com os 4 critérios cumpridos: botão ativado com animação de pulsação por 3 segundos.

---

### RF-011 — Confirmação de Fechamento com barreira textual de segurança

**Origem:** `Conforme RN-023 (Parte 01.2)`

O sistema deve exigir digitação da palavra "CONFIRMAR" em modal antes de registrar o Fechamento.

**Critério de aceite:**
- Clique em "Confirmar Fechamento": modal abre com resumo do caso (valor, partes, cenário, comissões).
- Campo de texto exige digitação exata de "CONFIRMAR" (maiúsculas, sem acento). Botão de confirmação no modal habilitado apenas após digitação correta.
- Após confirmação: caso muda para status Fechamento → Pós Fechamento imediatamente.
- Registros obrigatórios gravados: timestamp, nome do Analista, screenshot do estado dos 4 critérios, cópias dos documentos assinados, comprovante de depósito.
- QA: testar que "confirmar", "Confirmar", "CONFIRMAR " (com espaço) não habilitam o botão.

---

### RF-012 — Registro de anuência da construtora com SLA e aprovação

**Origem:** `Conforme RN-022.a (Parte 01.2)`, `Conforme RN-033 (Parte 01.2)`, `Conforme RN-146 (Parte 01.2)`

O sistema deve gerenciar o processo de obtenção e registro da anuência da construtora.

**Critério de aceite:**
- Analista registra solicitação com data, canal e contato responsável na construtora. Timestamp gravado.
- SLA de 15 dias corridos para resposta. No 10º dia: alerta ao Analista e Coordenador. No 15º dia: escalação automática ao Coordenador.
- Anuência recebida: Analista anexa documento, clica "Confirmar Anuência". Carimbo imutável aplicado.
- Exceção (contrato não exige anuência): Analista marca justificativa. Coordenador confirma em até 2 dias úteis. Critério marcado automaticamente como atendido.
- Casos com Valor Recuperado > R$500.000: confirmação da anuência exige validação adicional do Coordenador.
- Estados do critério: `Não solicitada` → `Solicitada` → `Recebida e verificada` | `Em atraso` | `Dispensada` | `Negada`.
- **Anuência negada formalmente pela construtora (RN-033):** Analista registra negativa no dossiê com documento comprobatório. Sistema move caso para status Cancelado com motivo "Anuência Negada". Conta Escrow inicia estorno integral automaticamente (conforme RF-008). Cedente e Cessionário notificados imediatamente. Histórico preservado no dossiê. [CORRIGIDO: PROBLEMA-002]

---

### RF-013 — Imutabilidade do Fechamento confirmado

**Origem:** `Conforme RN-024 (Parte 01.2)`

O sistema deve impedir qualquer reversão direta do Fechamento fora do fluxo formal de reversão.

**Critério de aceite:**
- Nenhum perfil (incluindo Master) possui botão "Desfazer Fechamento" na interface.
- API: tentativa de reverter status Fechamento diretamente retorna HTTP 403 com mensagem "Fechamento irreversível. Use o fluxo de reversão (RN-037)."
- Único caminho de desfazimento: fluxo de reversão dentro dos 15 dias corridos (RF-051 a RF-053).

## 3. Módulo: Negociação

### RF-014 — Submissão e validação automática de proposta

**Origem:** `Conforme RN-026 (Parte 01.2)`, `Conforme RN-028 (Parte 01.2)`, `Conforme RN-043 (Parte 01.2)`

O sistema deve validar automaticamente se a proposta atende ao valor mínimo do cenário antes de apresentá-la ao Cedente.

**Critério de aceite:**
- Proposta abaixo do valor mínimo do cenário: rejeitada automaticamente antes de chegar ao Analista. Cessionário notificado com motivo.
- Proposta válida: apresentada ao Analista no painel de Negociação em até 5 minutos `[DECISÃO AUTÔNOMA: SLA interno para triagem de proposta]`.
- Campos obrigatórios da proposta: valor ofertado, prazo de validade (72h), condição de pagamento (via Escrow), identidade do Cessionário.
- Rate limit: máximo 10 propostas por Cessionário por 24 horas. Tentativa além do limite: HTTP 429 com mensagem e tempo para reset.

---

### RF-015 — Apresentação da proposta ao Cedente com prazo de 72h

**Origem:** `Conforme RN-026 (Parte 01.2)`, `Conforme RN-047 (Parte 01.2)`

O sistema deve apresentar a proposta ao Cedente via painel e notificação em até 30 minutos da validação.

**Critério de aceite:**
- Proposta validada pelo sistema: notificação enviada ao Cedente via WhatsApp em até 30 minutos. Fallback SMS se WhatsApp falhar. Segundo fallback e-mail após 15 minutos sem entrega.
- Timer de 72h iniciado a partir do envio da notificação ao Cedente.
- No 48h: segundo lembrete enviado ao Cedente.
- Prazo expirado sem resposta: proposta expirada automaticamente. Status muda para "Expirada". Cessionário notificado.
- Se havia fila de Cessionários: próximo da fila é notificado automaticamente.

---

### RF-016 — Fluxo de contrapropostas com limite de 3 rodadas

**Origem:** `Conforme RN-026 (Parte 01.2)`, `Conforme RN-027 (Parte 01.2)`, `Conforme RN-045 (Parte 01.2)`

O sistema deve suportar até 3 rodadas de negociação antes de escalar ao Coordenador.

**Critério de aceite:**
- Contraproposta válida: histórico de lances imutável (append-only). Cada entrada registra: quem enviou, valor, timestamp.
- Após 3 rodadas sem acordo: caso escalado ao Coordenador automaticamente. Analista e Coordenador notificados.
- Histórico de negociação visível para Analista e Coordenador no painel do caso.
- Cedente aceita proposta em qualquer rodada: caso muda para Em Formalização imediatamente. Proposta muda para "Aceita". Propostas concorrentes para o mesmo caso mudam para "Superada".

---

### RF-017 — Rejeição de proposta com motivo obrigatório e notificação

**Origem:** `Conforme RN-143 (Parte 01.2)`

O sistema deve exigir motivo ao rejeitar uma proposta e notificar o Cessionário.

**Critério de aceite:**
- Campo "Motivo da Rejeição" obrigatório (mín. 20 caracteres) antes de confirmar rejeição.
- Cessionário notificado via WhatsApp em até 15 minutos da rejeição com o motivo fornecido pelo Cedente.
- Proposta rejeitada: status "Recusada". Imutável após aplicado.
- Cessionário pode submeter nova proposta após rejeição, respeitando rate limit (RF-014).

---

### RF-018 — Fila de Cessionários ordenada por score de viabilidade

**Origem:** `Conforme RN-046 (Parte 01.2)`, `Conforme RN-141 (Parte 01.2)`

O sistema deve manter fila de interesse ordenada por score calculado pelo Agente Analista de Oportunidades.

**Critério de aceite:**
- Cada caso em Oferta Ativa exibe aba "Fila de Cessionários" com lista ordenada por score.
- Score calculado pelo Agente IA (Analista de Oportunidades) — critérios: histórico de fechamentos, valor proposto, tempo de resposta histórico.
- Em caso de indisponibilidade do agente: fila ordenada por data de submissão (FIFO) como fallback.
- Fila visível para Analista, Coordenador e Master. Cessionário não vê a posição dos concorrentes.

### RF-019 — Aceitação de proposta com confirmação em 2 etapas

**Origem:** `Conforme RN-030 (Parte 01.2)`, `Conforme RN-142 (Parte 01.2)`

O sistema deve exigir confirmação em 2 etapas ao aceitar uma proposta.

**Critério de aceite:**
- 1ª etapa: Analista clica "Aceitar Proposta" e vê resumo financeiro (valor aceito, comissões estimadas, cenário).
- 2ª etapa: modal de confirmação exige clique adicional em "Confirmar Aceite". Botão cancelar disponível até este ponto.
- Após 2ª confirmação: caso muda para Em Formalização. Conta Escrow criada automaticamente (RF-004). Cedente e Cessionário notificados.
- Imutabilidade do aceite: após confirmação, aceite não pode ser desfeito pelo Analista. Somente cancelamento via Coordenador/Master.

---

### RF-020 — Escalação de negociação ao Coordenador após 3 rodadas

**Origem:** `Conforme RN-029 (Parte 01.2)`

O sistema deve escalar automaticamente ao Coordenador após 3 rodadas sem acordo.

**Critério de aceite:**
- 3ª rodada concluída sem aceite: sistema muda responsabilidade do caso para Coordenador automaticamente no módulo de Negociação.
- Analista original mantém visibilidade do caso mas perde capacidade de aceitar proposta.
- Coordenador recebe notificação urgente com histórico completo das 3 rodadas.
- Coordenador pode: mediar nova rodada (4ª+), aceitar proposta vigente, ou recomendar cancelamento.
- Log de escalação imutável no histórico do caso.

---

### RF-021 — Escalonamento de cenário descendente (D→C→B→A)

**Origem:** `Conforme RN-025 (Parte 01.2)`

O sistema deve suportar o escalonamento de cenário de retorno do Cedente.

**Critério de aceite:**
- Caso em Oferta Ativa há 30+ dias sem proposta: alerta "Estagnação" gerado no Pipeline para o Coordenador.
- Coordenador aciona sugestão de escalonamento: sistema envia simulação comparativa ao Cedente (valores do cenário atual vs. inferior).
- Cedente aceita: Termo de Aceite de Escalonamento enviado via ZapSign. Após assinatura: caso volta a Oferta Ativa com novo cenário. Timer SLA reinicia.
- Cedente recusa: caso permanece no cenário atual. Coordenador notificado.
- Escalonamento ascendente: proibido. Sistema não oferece opção de subir de cenário. Interface não exibe esta ação.
- Cedente já no Cenário A: botão de escalonamento não exibido.

---

### RF-022 — Conflito entre escalonamento e negociação ativa

**Origem:** `Conforme RN-025.a (Parte 01.2)`

O sistema deve gerenciar a fila do escalonamento quando há negociação ativa em paralelo.

**Critério de aceite:**
- Cedente solicita escalonamento com proposta ativa: escalonamento enfileirado. Cedente notificado da espera.
- Negociação resulta em aceite: escalonamento cancelado automaticamente.
- Negociação expira ou é recusada: escalonamento processado automaticamente sem intervenção manual.
- QA: testar o cenário de cancelamento automático do escalonamento após aceite de proposta.

---

### RF-023 — Painel de mediação com visão consolidada de propostas

**Origem:** `Conforme RN-048 (Parte 01.2)`

O sistema deve exibir painel de mediação com todas as propostas ativas de um caso.

**Critério de aceite:**
- Aba "Negociação" no painel do caso exibe todas as propostas com status individual.
- Para cada proposta: Cessionário (mascarado para Analista, completo para Coordenador+), valor, rodada atual, prazo restante, histórico de lances.
- Analista pode apresentar ao Cedente, aceitar, recusar ou fazer contraproposta a partir desta tela.
- Coordenador vê mesma tela com ações adicionais de mediação e recomendação de cancelamento.

## 4. Módulo: Formalização

### RF-024 — Checklist visual de critérios de Fechamento no painel de Formalização

**Origem:** `Conforme RN-032 (Parte 01.2)`, `Conforme RN-144 (Parte 01.2)`

O sistema deve exibir checklist visual dos 4 critérios de Fechamento atualizado em tempo real.

**Critério de aceite:**
- Tela de Formalização exibe 4 critérios como cards com status visual dinâmico.
- Atualização automática via Supabase Realtime ou polling de 10 segundos como fallback.
- Cada critério exibe: nome, status, quem cumprimentou, data/hora e link para evidência.
- Progress indicator global (ex: "2 de 4 critérios cumpridos") visível no topo do painel.

---

### RF-025 — Envio de envelopes ZapSign para assinatura

**Origem:** `Conforme RN-121 (Parte 01.5)`

O sistema deve enviar envelopes ZapSign com os documentos de cessão para assinatura eletrônica.

**Critério de aceite:**
- Analista clica "Enviar para Assinatura": sistema monta envelope com PDF, lista de signatários (nome, e-mail, CPF), ordem (Cedente → Cessionário → Construtora quando aplicável), prazo de 30 dias.
- POST à API ZapSign. Em caso de sucesso: `zapsign_document_token` persistido, URLs individuais salvas, status "Enviado" registrado.
- Sistema notifica cada signatário por e-mail com link individual de assinatura.
- Erro 4xx ZapSign: mensagem específica ao operador, sem registro de envio. Operador pode corrigir e reenviar.
- Erro 5xx/timeout ZapSign: banner de contingência exibido (RN-123). Caso permanece em Formalização. Modo manual disponível para Coordenador/Master.
- Card de assinaturas exibe progress (0/3, 1/3, 2/3, 3/3) e status individual de cada signatário.

---

### RF-026 — Recebimento e processamento de webhooks ZapSign

**Origem:** `Conforme RN-122 (Parte 01.5)`

O sistema deve processar webhooks ZapSign de forma validada e idempotente.

**Critério de aceite:**
- Validação HMAC-SHA256 obrigatória. Payload com assinatura inválida: descartado e registrado no audit trail.
- `document_signed` (parcial): status do signatário atualizado em tempo real no painel.
- `document_completed` (todos assinaram): critério "Instrumento assinado" marcado como cumprido. Analista notificado.
- `document_refused`: critério desmarcado, Analista notificado com urgência, motivo exibido.
- `document_expired`: critério desmarcado, Analista notificado para reenvio.
- Idempotência: mesmo `document_token` + mesmo evento processado apenas uma vez (sem duplicata).

### RF-027 — Prazo de 15 dias para depósito na Conta Escrow com notificações progressivas

**Origem:** `Conforme RN-034 (Parte 01.2)`, `Conforme RN-145 (Parte 01.2)`

O sistema deve gerenciar o prazo de depósito na Conta Escrow com notificações escalonadas ao Cessionário.

**Critério de aceite:**
- Prazo: 15 dias corridos a partir do aceite da proposta.
- Cronograma de notificações ao Cessionário: D+0 (instruções de depósito), D+5 (lembrete), D+10 (urgência), D+13 (alerta crítico), D+15 (cancelamento iminente).
- Analista e Coordenador alertados no D+10 e D+13.
- Depósito confirmado antes do prazo: prazo encerrado automaticamente. Sem notificações adicionais.

### RF-028 — Cancelamento automático por não-depósito no prazo

**Origem:** `Conforme RN-035 (Parte 01.2)`

O sistema deve cancelar automaticamente o caso se o depósito não for confirmado em 15 dias corridos após o aceite.

**Critério de aceite:**
- D+15 sem depósito confirmado: sistema executa cancelamento automático. Status muda para Cancelado. Motivo: "Prazo de depósito expirado".
- Cedente e Cessionário notificados imediatamente via WhatsApp/e-mail.
- Coordenador notificado com relatório do caso cancelado.
- Caso cancelado por esta regra: não pode ser reaberto. Cedente pode cadastrar novo caso para o mesmo imóvel.
- QA: verificar que jobs de verificação de prazo rodam corretamente às 00h (UTC-3) diariamente.

---

### RF-029 — Contingência ZapSign: assinatura manual

**Origem:** `Conforme RN-036 (Parte 01.2)`, `Conforme RN-123 (Parte 01.5)`

O sistema deve disponibilizar modo de contingência para assinatura manual quando ZapSign estiver indisponível.

**Critério de aceite:**
- Indisponibilidade ZapSign detectada (3 falhas consecutivas em 5 minutos): banner amarelo global no módulo Formalização com timestamp do início da indisponibilidade.
- Coordenador e Master veem botão "Registrar Assinatura Manual" no banner.
- Assinatura manual registrada: critério marcado como cumprido com nota "Assinatura manual registrada por [nome] em [data]". Documento físico deve ser digitalizado e anexado ao dossiê em até 24h.
- Restauração da integração: banner desaparece automaticamente. Toast: "Serviço de assinaturas restaurado."

---

## 5. Módulo: Reversão e Mediação

### RF-030 — Fluxo de reversão consensual dentro dos 15 dias

**Origem:** `Conforme RN-037 (Parte 01.2)`

O sistema deve suportar reversão consensual (ambas as partes concordam) dentro dos 15 dias pós-Fechamento.

**Critério de aceite:**
- Janela disponível: 15 dias corridos após o Fechamento, enquanto caso estiver em Pós-Fechamento.
- Ambas as partes solicitam reversão via painel: sistema move caso para Em Reversão. Conta Escrow congelada imediatamente.
- Gestor Financeiro processa estorno integral (RF-008). RS não recebe comissão.
- Após estorno confirmado: caso muda para Cancelado. Histórico preservado no dossiê.
- Após 15 dias sem solicitação: reversão bloqueada automaticamente. Botão removido da interface.

---

### RF-031 — Fluxo de mediação por desistência unilateral

**Origem:** `Conforme RN-038 (Parte 01.2)`

O sistema deve iniciar mediação quando apenas uma das partes solicita desistência.

**Critério de aceite:**
- Uma parte solicita desistência: caso muda para Em Mediação. Conta Escrow 100% congelada. Coordenador atribuído como mediador.
- SLA de mediação: 10 dias úteis para chegar a acordo.
- Acordo alcançado: caso encerrado conforme acordo (pode resultar em Cancelado ou retorno a fluxo normal `[DECISÃO AUTÔNOMA: retorno a Pós-Fechamento se acordo preserva o negócio]`).
- Sem acordo em 10 dias úteis: caso muda para Disputa Formal automaticamente.
- Valores retidos na Conta Escrow durante toda a mediação e disputa.

---

### RF-032 — Disputa Formal com retenção de valores

**Origem:** `Conforme RN-039 (Parte 01.2)`

O sistema deve gerenciar casos em Disputa Formal com valores retidos até resolução extrajudicial ou judicial.

**Critério de aceite:**
- Status Disputa Formal: apenas Master pode registrar movimentações e desfechos.
- Conta Escrow permanece congelada. Nenhuma distribuição ou estorno pode ser processado sem aprovação do Master.
- Master registra desfecho (extrajudicial ou judicial) com documento comprobatório.
- Desfecho registrado: caso muda para Cancelado com histórico completo.
- QA: validar que nenhum outro perfil pode executar ações financeiras em Disputa Formal.

---

## 6. Módulo: Cancelamento

### RF-033 — Cancelamento por perfil autorizado com motivo obrigatório

**Origem:** `Conforme RN-040 (Parte 01.2)`

O sistema deve permitir cancelamento por Coordenador e Master com motivo obrigatório em qualquer estado antes do Fechamento.

**Critério de aceite:**
- Cancela em estado pré-Formalização: Coordenador ou Master pode cancelar. Campo motivo obrigatório (mín. 30 caracteres) `[DECISÃO AUTÔNOMA: 30 chars para garantir justificativa mínima]`.
- Cancela em Em Formalização: exige aprovação adicional do Master se caso tiver depósito confirmado na Conta Escrow.
- Após Fechamento: cancelamento não disponível. Somente fluxo de reversão (RF-030 a RF-032).
- Cancelamento confirmado: Cedente e Cessionário notificados. Conta Escrow estornada se havia depósito.

---

### RF-034 — Proibição de reativação de caso cancelado

**Origem:** `Conforme RN-041 (Parte 01.2)`

O sistema deve impedir que qualquer caso com status Cancelado seja reativado.

**Critério de aceite:**
- Interface: botão "Reativar Caso" não existe para nenhum perfil.
- API: PATCH/PUT em caso com status Cancelado retorna HTTP 409 com mensagem "Caso cancelado não pode ser reativado."
- Cedente pode cadastrar novo caso para o mesmo imóvel após cancelamento (RN-003 exceto casos ativos).

---

### RF-035 — Histórico de cancelamento imutável no dossiê

**Origem:** `Conforme RN-042 (Parte 01.2)`

O sistema deve preservar o histórico completo do caso mesmo após cancelamento.

**Critério de aceite:**
- Caso cancelado: todos os documentos, propostas, comunicações e registros de auditoria preservados permanentemente no dossiê.
- Dossiê acessível para Coordenador e Master em modo somente leitura após cancelamento.
- Deleção (mesmo soft delete) de qualquer registro do dossiê de caso cancelado: proibida por RLS no banco de dados.

---

## 7. Módulo: Integrações Celcoin (Conta Escrow)

### RF-036 — Integração Celcoin: criação de Conta Escrow

**Origem:** `Conforme RN-124 (Parte 01.5)`

O sistema deve criar Conta Escrow via API Celcoin ao caso entrar em Formalização.

**Critério de aceite:**
- POST à API Celcoin com dados do caso ao mudar status.
- Resposta de sucesso: ID da conta Celcoin persistido, instruções de depósito geradas e enviadas ao Cessionário.
- Falha na API: sistema registra no audit trail, exibe banner de contingência, Analista recebe notificação para ação manual.

---

### RF-037 — Integração Celcoin: confirmação de depósito via webhook

**Origem:** `Conforme RN-125 (Parte 01.5)`, `Conforme RN-126 (Parte 01.5)`

O sistema deve processar webhooks Celcoin de confirmação de depósito com validação HMAC-SHA256.

**Critério de aceite:**
- Validação HMAC-SHA256 obrigatória. Falha na validação: payload descartado, tentativa registrada no audit trail.
- Webhook de depósito confirmado: status da Conta Escrow atualizado em até 30 segundos. Analista notificado em tempo real.
- Idempotência: mesmo evento de depósito processado apenas uma vez.

---

### RF-038 — Integração Celcoin: distribuição e estorno

**Origem:** `Conforme RN-127 (Parte 01.5)`

O sistema deve executar distribuições e estornos via API Celcoin.

**Critério de aceite:**
- Distribuição: API chamada com breakdown detalhado (Cedente, RS, crédito residual). Comprovantes gerados para todas as partes.
- Estorno: API chamada com valor integral. Comprovante gerado para o Cessionário.
- Falha em distribuição ou estorno: sistema pausa operação, registra no audit trail, notifica Gestor Financeiro e Master imediatamente.

---

### RF-039 — Modo de contingência Celcoin

**Origem:** `Conforme RN-128 (Parte 01.5)`

O sistema deve ter modo de contingência documentado quando Celcoin estiver indisponível.

**Critério de aceite:**
- Indisponibilidade Celcoin: banner vermelho global no módulo Financeiro e Formalização com timestamp.
- Operações afetadas (criação de conta, confirmação de depósito, distribuição, estorno) ficam pausadas.
- Master pode registrar operação manual com upload de comprovante externo.
- Restauração: banner removido automaticamente. Jobs pendentes reprocessados.
