## 1. Módulo: Triagem

### RF-040 — Fila de triagem ordenada por FIFO com alerta de seleção fora de ordem

**Origem:** `Conforme RN-055 (Parte 01.3)`

O sistema deve exibir fila de triagem ordenada por data de entrada (FIFO) e alertar o Analista ao selecionar um caso fora de ordem.

**Critério de aceite:**
- Fila ordenada por data de entrada (caso mais antigo no topo). Cada item exibe: ID, endereço, data de entrada, tempo na fila, progresso do dossiê (X/6 documentos).
- Caso mais antigo: borda esquerda colorida azul + label "Próximo na fila".
- Seleção fora de ordem (não é o mais antigo): banner amarelo no topo do painel com mensagem e botão "Ir para o caso mais antigo". Seleção permitida mas registrada no audit trail.
- Fila vazia: estado vazio com ícone de checklist + "Todos os casos foram triados."
- QA: verificar que o log de auditoria registra seleções fora de ordem com timestamp e identificação do Analista.

### RF-041 — Verificação de documento com carimbo imutável

**Origem:** `Conforme RN-056 (Parte 01.3)`

O sistema deve registrar carimbo imutável ao verificar cada documento do dossiê.

**Critério de aceite:**
- Analista clica "Verificar": sistema registra carimbo (nome, data, hora) imediatamente. Carimbo não pode ser removido por nenhum perfil.
- Animação de transição ⏳ → ✅ em 0,3s ao verificar. Animação ⏳ → ❌ ao rejeitar.
- Carimbo exibido como tooltip ao passar cursor sobre ícone ✅: "Verificado por [Nome] em [data] às [hora]."
- Rejeição: exige campo de justificativa (mín. 10 caracteres) antes de confirmar.
- QA: tentativa de PATCH no carimbo via API deve retornar HTTP 403.

---

### RF-042 — Reenvio de documento rejeitado com histórico preservado

**Origem:** `Conforme RN-148 (Parte 01.3)`

O sistema deve aceitar reenvio de documento rejeitado mantendo o histórico do documento anterior.

**Critério de aceite:**
- Cedente reenvia documento: novo arquivo substitui o anterior na checklist ativa. Status do item muda de ❌ para ⏳.
- Documento anterior não é apagado: histórico acessível na aba "Histórico" do dossiê com arquivo rejeitado, motivo, carimbo e data/hora.
- Analista deve re-verificar o novo documento (carimbo anterior não transfere).
- QA: verificar que após reenvio, histórico exibe 2 entradas para o mesmo tipo documental.

### RF-043 — Salvamento automático do progresso de triagem

**Origem:** `Conforme RN-057 (Parte 01.3)`

O sistema deve salvar automaticamente o progresso de triagem quando o Analista navegar para outra tela.

**Critério de aceite:**
- Analista navega para outra tela com triagem parcial: progresso salvo automaticamente. Sem notificação ao Analista.
- Retorno ao caso: painel exibe "Verificação em andamento — X de 6 documentos revisados." Estado idêntico ao deixado.
- Documentos verificados mantêm ✅. Documentos rejeitados mantêm ❌. Pendentes mantêm ⏳.

### RF-044 — Bloqueio de caso por inadimplência com sugestão automática de cancelamento

**Origem:** `Conforme RN-058 (Parte 01.3)`, `Conforme RN-001 (Parte 01.1)`, `Conforme RN-077 (Parte 01.3)`

O sistema deve bloquear caso por inadimplência e sugerir cancelamento após 60 dias sem ação do Cedente.

**Critério de aceite:**
- Analista clica "Bloquear Caso": modal de confirmação exibido. Confirmado: status muda para Bloqueado. Cedente notificado com lista de parcelas pendentes e botão "Enviar Comprovante".
- Banner fixo vermelho no topo do painel: "Caso Bloqueado — Inadimplência". Todas as ações de avanço desabilitadas.
- 60 dias sem ação do Cedente: alerta automático ao Coordenador com sugestão de cancelamento. Coordenador decide.
- Desbloqueio: exige checklist do motivo original resolvido. Comprovantes novos devem estar no dossiê. Apenas Analista atribuído ou Coordenador pode desbloquear.
- QA: verificar job diário que verifica casos bloqueados há 60 dias.

### RF-045 — Checklist de dossiê com barra de progresso e aprovação de triagem

**Origem:** `Conforme RN-002 (Parte 01.1)`

O sistema deve exibir checklist dos 6 documentos obrigatórios com barra de progresso e controlar o botão "Aprovar Triagem".

**Critério de aceite:**
- Barra de progresso: "X de 6 documentos verificados" com preenchimento proporcional.
- Botão "Aprovar Triagem" desabilitado enquanto houver pendências. Tooltip: "Faltam X documentos para habilitar a aprovação."
- Todos os 6 documentos verificados: botão "Aprovar Triagem" habilitado.
- Botão "Solicitar Documentos": gera preview do e-mail antes de enviar. Lista exata dos documentos pendentes para o Cedente.
- Aprovação: caso muda para Qualificado. Carimbo do Analista com data/hora gravado.
- Sem aprovação condicional: nenhum perfil pode aprovar com menos de 6 documentos.

### RF-046 — Exceção de adimplência com aprovação do Coordenador e assinatura digital

**Origem:** `Conforme RN-001.a (Parte 01.1)`, `Conforme RN-148 (Parte 01.3)`

O sistema deve suportar exceção de adimplência mediante confirmação da construtora e aprovação do Coordenador com assinatura digital.

**Critério de aceite:**
- Analista registra exceção: documento da construtora anexado ao dossiê.
- Solicitação de aprovação enviada ao Coordenador automaticamente.
- Coordenador aprova com assinatura digital (via ZapSign ou mecanismo equivalente) `[DECISÃO AUTÔNOMA: assinatura digital para exceção de adimplência garante rastreabilidade jurídica da decisão]`.
- Após aprovação: caso muda para Qualificado. Exceção registrada no dossiê com carimbo de ambos (Analista + Coordenador).
- Coordenador não aprova: caso permanece Bloqueado.
- QA: tentativa de aprovação de exceção sem assinatura deve retornar erro de validação.

### RF-047 — Unicidade de caso ativo por imóvel e Cedente

**Origem:** `Conforme RN-003 (Parte 01.1)`, `Conforme RN-003.a (Parte 01.1)`

## 2. Módulo: SLAs

### RF-048 — Monitoramento contínuo de SLA com alertas automáticos

**Origem:** `Conforme RN-059 (Parte 01.3)`, `Conforme RN-062 (Parte 01.3)`

O sistema deve monitorar continuamente os SLAs de todos os casos ativos e gerar alertas por etapa.

**Critério de aceite:**
- Badge de SLA em todos os cards de caso: verde (dentro do SLA), amarelo (≤20% do prazo máximo restante), vermelho (estourado).
- Badge acessível: ícone diferenciado (check/relógio/exclamação) + label textual. Não dependente apenas de cor.
- Tooltip: "SLA: X de Y dias. Etapa: [nome]."
- SLA ≤20%: alerta no painel do Analista e Coordenador (somente painel).
- SLA estourado: alerta por e-mail + painel ao Coordenador e Master.
- Sistema não cancela caso automaticamente por SLA estourado. Ação corretiva é manual.
- QA: verificar que job de monitoramento de SLA roda a cada 15 minutos `[DECISÃO AUTÔNOMA: 15 minutos para monitoramento de SLA]`.

### RF-049 — SLAs configuráveis pelo Master

**Origem:** `Conforme RN-110 (Parte 01.4)`

O sistema deve permitir ao Master configurar os prazos de SLA por etapa sem deploy de código.

**Critério de aceite:**
- Menu Configurações → SLAs: tabela editável com SLA Alvo e SLA Máximo por transição.
- Alterações afetam apenas novos casos. Casos existentes usam o SLA vigente no momento de sua criação (snapshot de configuração, conforme RN-111).
- Pré-visualização obrigatória antes de salvar. Log de alteração registrado automaticamente no audit trail.
- Valores inválidos (ex: SLA Alvo > SLA Máximo): erro de validação inline.

### RF-050 — Reinício de SLA após escalonamento e pausa por indisponibilidade

**Origem:** `Conforme RN-060 (Parte 01.3)`, `Conforme RN-061 (Parte 01.3)`

O sistema deve reiniciar o timer de SLA após escalonamento aceito e pausar durante indisponibilidades de integração.

**Critério de aceite:**
- Escalonamento aceito + ZapSign assinado: timer de "Oferta Ativa → Em Negociação" reiniciado no momento da assinatura. Alerta de estagnação marcado como resolvido.
- Indisponibilidade de integração crítica detectada (3 falhas em 5 minutos): SLA de Formalização pausado automaticamente. Timestamp de início de pausa registrado.
- Integração restaurada: timer retoma a contagem. Duração da pausa descontada do prazo total.
- QA: simular indisponibilidade de 1 hora e verificar que SLA não inclui esse período.

## 3. Módulo: Notificações Automáticas

### RF-051 — Envio de notificações por e-mail e painel interno com SLA de entrega

**Origem:** `Conforme RN-062 (Parte 01.3)`, `Conforme RN-063 (Parte 01.3)`

O sistema deve enviar notificações para todos os eventos do ciclo de vida do caso via e-mail e painel interno.

**Critério de aceite:**
- Painel interno: notificação exibida em até 30 segundos do evento.
- E-mail: disparado em até 5 minutos do evento via RabbitMQ (fila assíncrona com retry).
- Todos os 30 eventos da tabela RN-063 implementados com destinatário e canal corretos.
- Centro de notificações: ícone de sino com badge numérico. Painel dropdown com lista cronológica. Cada notificação clicável (redireciona ao contexto). Botão "Marcar todas como lidas".
- Notificações não lidas: fundo levemente mais escuro + ponto azul na lateral.

---

### RF-052 — Notificações via WhatsApp (canal primário) e SMS (fallback)

**Origem:** `Conforme RN-063 (Parte 01.2)`, `Conforme RN-064 (Parte 01.2)`, `Conforme RN-065 (Parte 01.2)`

O sistema deve enviar notificações ao Cedente e Cessionário via WhatsApp com fallback para SMS e e-mail.

**Critério de aceite:**
- Canal primário: WhatsApp via Meta Cloud API. Templates pré-aprovados para cada tipo de evento.
- Falha no WhatsApp após 3 tentativas em 5 minutos: sistema tenta SMS via Twilio automaticamente.
- Falha no SMS após 3 tentativas: sistema envia e-mail como segundo fallback.
- Todas as tentativas (sucesso ou falha) registradas no log de notificações do caso.
- Operador do Admin não pode desabilitar notificações de eventos críticos (depósito, fechamento, cancelamento).

---

### RF-053 — Gestão de falhas de entrega de notificação

**Origem:** `Conforme RN-064 (Parte 01.3)`

O sistema deve registrar falhas de entrega e permitir reenvio após correção do e-mail.

**Critério de aceite:**
- E-mail inválido ou bounce: falha registrada no log com motivo. Alerta ao Coordenador no painel.
- Log de notificações: badge "Falhou" em vermelho. Coordenador vê motivo e botão "Corrigir E-mail" (abre perfil do usuário).
- Após correção: botão "Reenviar" ativo ao lado da notificação falhada.
- Cada notificação registrada com: evento, destinatário, canal, data/hora, status de entrega.

### RF-055 — Suspensão, reativação e inatividade automática de usuário pelo Master

**Origem:** `Conforme RN-069 (Parte 01.3)`, `Conforme RN-066 (Parte 01.3)` [CORRIGIDO: PROBLEMA-004], `Conforme RN-067 (Parte 01.3)` [CORRIGIDO: PROBLEMA-005] [DECISÃO APLICADA: DEC-002]

O sistema deve permitir ao Master suspender e reativar usuários com justificativa obrigatória, aplicando impactos cascata conforme o tipo de usuário, e gerenciar inatividade automática por tempo.

**Critério de aceite:**
- Suspensão: campo motivo obrigatório (mín. 20 caracteres). Usuário suspenso: acesso bloqueado imediatamente. Todas as sessões ativas encerradas.
- Reativação: Master reativa com justificativa. Usuário reativado pode fazer login normalmente.
- Operador suspenso não pode ser atribuído a novos casos. Casos existentes redirecionados ao Coordenador para reatribuição.
- Suspensão de Master: somente se houver outro Master ativo (RN-117). Sistema bloqueia se for o único.
- **Impacto cascata — suspensão de Cedente (RN-066):** todos os casos ativos do Cedente ficam congelados imediatamente. Badge "Caso Congelado — Cedente suspenso" exibido no Pipeline. Analistas responsáveis notificados. Casos reabertos ao reativar o Cedente.
- **Impacto cascata — suspensão de Cessionário (RN-066):** todas as propostas ativas do Cessionário canceladas automaticamente com motivo "Cessionário suspenso". Analistas dos casos afetados notificados com lista de propostas canceladas.
- **Modal de confirmação com resumo de impacto (RN-066):** ao clicar "Suspender", sistema exibe: "Suspender [nome] afetará: [X] casos ativos (congelados) e [Y] propostas ativas (canceladas)." Botão "Confirmar Suspensão" em vermelho. Após confirmação: toast "Usuário suspenso. [X] casos congelados e [Y] propostas canceladas."
- **Inatividade automática (RN-067):** job diário verifica usuários com último acesso há 180+ dias corridos. Status muda automaticamente para "Inativo". Usuário Inativo ao fazer login: status reativado automaticamente para "Ativo" sem perda de dados ou histórico.
- QA: suspender Cedente com 2 casos ativos → verificar que ambos ficam com badge "Caso Congelado". Suspender Cessionário com 1 proposta ativa → verificar cancelamento automático. Simular usuário sem acesso há 181 dias → verificar status "Inativo" no job diário.

### RF-068 — Comentários internos por caso

**Origem:** `Conforme RN-086 (Parte 01.3)`

O sistema deve suportar comentários internos por caso, visíveis apenas para operadores.

**Critério de aceite:**
- Aba "Comentários" no painel do caso. Campo de texto com envio.
- Comentários visíveis apenas para Analista, Coordenador, Gestor Financeiro e Master. Cedente e Cessionário nunca veem estes comentários.
- Soft delete: Analista pode "remover" o próprio comentário mas ele permanece no audit trail.
- Comentário marcado como removido exibe "Comentário removido por [nome] em [data]" em vez de sumir.

## 7. Módulo: Autenticação e Perfis

### RF-069 — Login com 2FA obrigatório para todos os operadores

**Origem:** `Conforme RN-005 (Parte 01.1)`, `Conforme RN-004 (Parte 01.1)`

O sistema deve exigir autenticação de dois fatores (TOTP) para todos os operadores do Admin.

**Critério de aceite:**
- Primeiro acesso: operador obrigado a configurar aplicativo autenticador (TOTP). Sem exceção.
- Login padrão: e-mail + senha → código TOTP de 6 dígitos gerado pelo autenticador.
- TOTP inválido ou expirado: login recusado com mensagem "Código inválido. Tente novamente." Tentativa registrada.
- QA: verificar que endpoint de login sem TOTP retorna HTTP 401 mesmo com e-mail e senha corretos.

### RF-070 — Bloqueio de conta após 5 tentativas com desbloqueio em 30 minutos

**Origem:** `Conforme RN-006 (Parte 01.1)`

O sistema deve bloquear a conta após 5 tentativas de login malsucedidas por 30 minutos.

**Critério de aceite:**
- 5ª tentativa malsucedida: conta bloqueada por 30 minutos. Mensagem "Conta bloqueada por excesso de tentativas. Tente novamente em [horário]."
- Desbloqueio automático após 30 minutos (sem intervenção manual).
- Master pode desbloquear manualmente antes dos 30 minutos via painel de Usuários.
- Cada tentativa registrada no audit trail com timestamp e IP.
- Rate limiting via @nestjs/throttler: 5 tentativas / 30 minutos por IP + por conta.

---

### RF-071 — Sessões JWT com access token 15min e refresh token 8h

**Origem:** `Conforme RN-007 (Parte 01.1)`, `Conforme RN-134 (Parte 01.5)`

O sistema deve implementar JWT com access token de 15 minutos e refresh token de 8 horas.

**Critério de aceite:**
- Access token: validade de 15 minutos. Armazenado em memória (Zustand). Não persiste em localStorage.
- Refresh token: validade de 8 horas. Armazenado em httpOnly cookie (web) ou SecureStore (mobile).
- Access token expirado: sistema renova automaticamente com o refresh token (silent refresh). Sem logout forçado.
- Refresh token expirado: usuário redirecionado para login. Notificação in-app: "Sua sessão expirou. Faça login novamente." (RN-140).
- Logout: access token e refresh token invalidados. Refresh token blacklistado no Redis.

### RF-072 — Gestão de sessões ativas pelo operador

**Origem:** `Conforme RN-139 (Parte 01.1)`

O sistema deve permitir que o operador visualize e encerre suas próprias sessões ativas.

**Critério de aceite:**
- Menu de perfil do operador: aba "Sessões Ativas" com lista de sessões (dispositivo, IP, última atividade).
- Botão "Encerrar esta sessão" por sessão individual. Botão "Encerrar todas as outras sessões".
- Encerramento: refresh token blacklistado no Redis imediatamente.

## 8. Módulo: Onboarding

### RF-073 — Tour guiado obrigatório para novos operadores

**Origem:** `Conforme RN-008 (Parte 01.1)`

O sistema deve exibir tour guiado interativo no primeiro acesso de cada novo operador.

**Critério de aceite:**
- Primeiro login após convite: tour guiado iniciado automaticamente. Não pode ser ignorado no primeiro acesso.
- Tour personalizado por perfil: Analista tour cobre Triagem, Pipeline, Notificações. Coordenador tour cobre Pipeline, Escalação, Aprovações. Gestor tour cobre Financeiro. Master tour cobre Configurações + todos.
- Progresso do tour salvo: se operador fechar o browser no meio do tour, ele retoma de onde parou no próximo acesso.
- Após completar: botão "Repetir tour" disponível no menu de ajuda.
- QA: verificar que tour não reexibe após conclusão no primeiro acesso.

## 9. Módulo: Painel do Caso (360°)

### RF-074 — Painel consolidado do caso com 6 abas

**Origem:** `Conforme RN-080 (Parte 01.3)`, `Conforme RN-081 a RN-086 (Parte 01.3)`

O sistema deve exibir painel consolidado do caso com abas para todas as informações relevantes.

**Critério de aceite:**
- 6 abas obrigatórias: Dossiê (RN-081), Timeline (RN-082), Propostas (RN-083), Financeiro (RN-084), Comunicação (RN-085), Comentários (RN-086).
- Aba ativa persistida na URL (query param `?tab=dossie`). Compartilhável.
- Aba Dossiê: checklist de 6 documentos com status e histórico de versões.
- Aba Timeline: histórico imutável de transições de estado.
- Aba Propostas: todas as propostas com status e histórico de lances.
- Aba Financeiro: Conta Escrow, comissões calculadas, breakdown de distribuição.
- Aba Comunicação: histórico de notificações enviadas às partes para este caso.
- Aba Comentários: comentários internos apenas para operadores.
- Cada aba acessível por perfil conforme permissões: Gestor Financeiro não vê Aba Triagem.
