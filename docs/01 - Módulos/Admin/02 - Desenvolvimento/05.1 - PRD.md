## 1. Problema e Contexto

### 1.1 O Problema

O mercado de cessão de contratos imobiliários no Brasil carece de uma plataforma estruturada de intermediação. Cedentes que querem transferir seu contrato com a construtora enfrentam:

1. **Falta de comprador qualificado:** sem uma plataforma de match, o Cedente desconhece Cessionários interessados.
2. **Risco financeiro elevado:** sem garantia de pagamento (escrow), o Cedente fica exposto a inadimplência.
3. **Complexidade documental:** anuência da construtora, instrumento de cessão, tabela de preço — processos manuais e passíveis de erro.
4. **Ausência de rastreabilidade:** sem histórico auditável, disputas são difíceis de resolver.

O Repasse Seguro resolve todos esses problemas em uma única plataforma: match entre Cedentes e Cessionários, gestão de documentos, Conta Escrow garantida e operação interna rastreável.

### 1.2 Contexto Estratégico

O Módulo Admin é a **espinha dorsal operacional** do Repasse Seguro. Enquanto os módulos públicos (Cedente, Cessionário) gerenciam a experiência dos clientes finais, o Admin é onde os operadores internos executam, monitoram e configuram toda a operação.

**Sem o Admin, não há operação.** Cada caso passa obrigatoriamente por triagem, negociação, formalização e fechamento — todos gerenciados pelo Admin.

## 2. Público-Alvo e Personas

### 2.1 Personas Primárias (Operadores Internos)

**Persona 1 — Analista**

| **Atributo** | **Descrição** |
|---|---|
| **Função** | Executa a operação diária: triagem, negociação, formalização e fechamento dos casos atribuídos |
| **Volume de trabalho** | Gerencia de 5 a 20 casos simultâneos (estimativa — `[DECISÃO AUTÔNOMA]`) |
| **Ponto de dor** | Perder prazo de SLA sem aviso. Não saber o próximo passo em um caso complexo. |
| **Necessidade central** | Visibilidade clara do status de cada caso e alertas proativos de SLA |
| **Acesso** | Dashboard, Pipeline (seus casos), Triagem, Negociação, Formalização |

**Persona 2 — Coordenador**

| **Atributo** | **Descrição** |
|---|---|
| **Função** | Supervisiona Analistas, atribui casos, aprova escalonamentos, resolve disputas |
| **Volume de trabalho** | Supervisiona equipe de 3 a 10 Analistas (estimativa — `[DECISÃO AUTÔNOMA]`) |
| **Ponto de dor** | Visão fragmentada de tudo que está acontecendo. Escalações chegando sem contexto. |
| **Necessidade central** | Visão 360° de todos os casos e fluxo claro de escalações |
| **Acesso** | Todos os módulos exceto Configurações. Financeiro em somente leitura. |

**Persona 3 — Gestor Financeiro**

| **Atributo** | **Descrição** |
|---|---|
| **Função** | Monitora Conta Escrow, processa estornos, inicia reversões, exporta relatórios financeiros |
| **Volume de trabalho** | Monitora todas as Contas Escrow ativas simultaneamente |
| **Ponto de dor** | Distribuição executada com valor errado. Depósito confirmado após prazo. |
| **Necessidade central** | Monitoramento em tempo real com alertas de depósitos pendentes e prazos |
| **Acesso** | Dashboard (financeiro), Pipeline (leitura), Financeiro, Usuários (leitura), Relatórios (Receita+Auditoria) |

**Persona 4 — Master**

| **Atributo** | **Descrição** |
|---|---|
| **Função** | Acesso irrestrito. Configura parâmetros globais, gerencia operadores, aprova ações críticas |
| **Volume de trabalho** | Foco em configuração, aprovações críticas e auditoria — não executa triagem diária |
| **Ponto de dor** | Parâmetros configurados de forma errada afetando cálculos de comissão em cascata |
| **Necessidade central** | Controle total com trilha de auditoria de cada alteração de configuração |
| **Acesso** | Tudo |

## 3. Objetivos e Métricas de Sucesso

### 3.1 Objetivos Estratégicos

| **Objetivo** | **Por quê** |
|---|---|
| Zero casos com SLA estourado sem alerta | Cada estouro de SLA representa atraso no faturamento |
| 100% dos fechamentos com 4 critérios cumpridos | Critério inegociável de compliance financeiro |
| Rastreabilidade total de cada ação de cada operador | Auditabilidade legal e resolução de disputas |
| Configuração de parâmetros sem deploy de código | Agilidade operacional — Master não depende de engenharia |
| Supervisão humana de agentes de IA com takeover em 1 clique | Controle sobre automações críticas |

## 4. Escopo Completo

### 4.1 O que está no escopo

O Módulo Admin cobre 100% da operação interna do Repasse Seguro:

1. **Autenticação e Controle de Acesso** — login, 2FA, perfis, sessões (RN-004 a RN-007, RN-134, RN-139, RN-140)
2. **Onboarding** — tour guiado para novos operadores (RN-008)
3. **Dashboard** — KPIs em tempo real por perfil (RN-009 a RN-013)
4. **Pipeline** — visão kanban/lista de todos os casos (RN-014 a RN-017, RN-147, RN-151)
5. **Triagem** — verificação documental e adimplência (RN-001/002/003, RN-055 a RN-058, RN-077, RN-148)
6. **Negociação** — propostas, contrapropostas, escalonamento (RN-024 a RN-031, RN-043 a RN-048, RN-141 a RN-143)
7. **Formalização** — ZapSign, anuência, depósito Escrow (RN-032 a RN-036, RN-144 a RN-146)
8. **Fechamento** — 4 critérios + distribuição automática (RN-022, RN-023)
9. **Reversão/Mediação/Disputa** — pós-fechamento (RN-037 a RN-039)
10. **Cancelamento** — por qualquer estado (RN-040 a RN-042)
11. **Comissões e Conta Escrow** — cálculo por cenário e distribuição (RN-018 a RN-021, RN-049 a RN-054)
12. **Módulo Financeiro** — monitoramento, bloqueio, conciliação (RN-087 a RN-091, RN-149)
13. **Supervisão IA** — painel dos agentes, takeover, logs (RN-093 a RN-098)
14. **Gestão de Usuários** — Cedentes, Cessionários, operadores (RN-067 a RN-071, RN-115)
15. **Relatórios** — 6 tipos com exportação (RN-099 a RN-108)
16. **Configurações** — parâmetros globais pelo Master (RN-109 a RN-120)
17. **Integrações externas** — ZapSign, Celcoin, WhatsApp, SMS (RN-121 a RN-128, RN-138, RN-150)
18. **Transversais** — audit trail, lock otimista, LGPD, idempotency, registro de transições de status (RN-016, RN-129 a RN-137) [CORRIGIDO: PROBLEMA-010]

## 6. Restrições e Dependências Globais

### 6.1 Restrições Técnicas

| **Restrição** | **Origem** |
|---|---|
| Next.js proibido — dashboard 100% logado → React + Vite (SPA) | Doc 02 Stacks, seção 3.1 |
| TypeScript `strict: true` em todo o código | Doc 02 Stacks, seção 2.1 |
| Axios proibido — usar fetch nativo com wrapper | Doc 02 Stacks, seção 5 |
| Auto-increment proibido em PKs — UUID v4 | Doc 02 Stacks, seção 2.3 |
| `$db.Timestamp` sem timezone proibido | Doc 02 Stacks, seção 2.3 |
| PostgreSQL auto-gerenciado proibido em produção — Supabase obrigatório | Doc 02 Stacks, seção 2.1 |
| ORM único — Prisma 6+ | Doc 02 Stacks, seção 2.1 |

### 6.2 Dependências Externas

| **Serviço** | **Criticidade** | **Contingência** |
|---|---|---|
| ZapSign (assinatura eletrônica) | Alta — critério de Fechamento | Assinatura manual via Coordenador/Master (RN-036, RN-123) |
| Celcoin (Conta Escrow) | Crítica — sem ela o Fechamento é inviável | Modo de contingência documentado (RN-128) |
| Meta Cloud API (WhatsApp) | Alta — canal primário de notificação | Fallback para SMS via Twilio (RN-064) |
| Twilio (SMS) | Média — fallback do WhatsApp | E-mail como segundo fallback (RN-065) |
| Supabase (BaaS) | Crítica — banco + auth + realtime | Sem contingência — SLA Supabase ≥ 99,9% |
