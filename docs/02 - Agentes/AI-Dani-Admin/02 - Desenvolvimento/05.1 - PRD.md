## 1. Visão do Produto

### 1.1 Problema

A Repasse Seguro opera agentes de IA (Dani-Cessionário, Dani-Cedente) em contato direto com usuários. Sem supervisão centralizada, incidentes operacionais (respostas incorretas, confiança baixa, erros de API, manipulação por prompt injection) não podem ser detectados em tempo real, investigados com evidências ou corrigidos antes de impactar mais usuários.

### 1.2 Solução

O módulo **AI-Dani-Admin** fornece à equipe Admin da Repasse Seguro:
1. Visibilidade total sobre todas as interações dos agentes (monitoramento contínuo).
2. Capacidade de intervenção manual (takeover) quando a qualidade da IA não é adequada.
3. Dashboard de métricas consolidadas para decisões de gestão baseadas em dados.
4. Controle de parâmetros operacionais (threshold de confiança) sem deploy de código.
5. Alertas automáticos para incidentes críticos via Slack e e-mail.
6. Gates de segurança que impedem lançamento de agentes sem pré-requisitos atendidos.

### 1.3 Usuário-Alvo

| Perfil | Descrição |
|---|---|
| **Admin** | Equipe operacional da Repasse Seguro. Acesso exclusivo a todos os componentes deste módulo. |
| **Cessionário / Cedente** | Não têm acesso a nenhum componente do módulo AI-Dani-Admin. Acesso bloqueado em 100% das operações. |

## 2. Requisitos Funcionais

### Módulo 1 — Supervisão de Interações (Painel Supervisão IA)

**RF-001** ← RN-DA-030
O sistema deve fornecer ao Admin uma lista de todas as interações dos agentes (Dani-Cessionário e Dani-Cedente), com os seguintes campos por interação:
- Identificação do usuário: anonimizada na listagem, detalhada na visualização individual.
- Data e hora da interação (timestamp com timezone America/Fortaleza).
- Pergunta enviada pelo usuário.
- Resposta gerada pelo agente.
- Nível de confiança da resposta (percentual de 0 a 100%).
- Dados utilizados para gerar a resposta.
- Latência da resposta (segundos).
- Identificador do agente (Dani-Cessionário, Dani-Cedente, Dani-Admin) — filtrável.

**Critério de aceite RF-001:**
- Dado: Admin autenticado acessa o painel de Supervisão IA.
- Quando: A tela carrega.
- Então: Lista exibe todas as interações com os 8 campos acima preenchidos. Identificação do usuário exibe ID anonimizado (não nome/CPF). Identidade do agente é visível por filtro ou coluna.

---

**RF-002** ← RN-DA-030
O sistema deve exibir estado vazio com ícone ilustrativo e texto "Nenhuma interação registrada no período selecionado." quando não houver interações. O estado vazio deve incluir a sugestão: "Tente ajustar o período ou os filtros aplicados."

**Critério de aceite RF-002:**
- Dado: Admin acessa o painel com filtro que retorna zero interações.
- Quando: A lista é exibida.
- Então: Ícone ilustrativo + texto de estado vazio + sugestão de ajuste são exibidos. Nenhum elemento da lista de dados é renderizado.

---

**RF-003** ← RN-DA-030
O sistema deve suportar filtros na lista de interações: por data/período, por usuário (via ID anonimizado), por nível de confiança (range ou abaixo de threshold) e por agente (Dani-Cessionário, Dani-Cedente). Filtros ativos são exibidos como chips removíveis. O Admin pode remover filtros individualmente (x no chip) ou todos com botão "Limpar filtros". A lista exibe skeleton loading inline durante a aplicação do filtro — sem substituir a tela inteira.

**Critério de aceite RF-003:**
- Dado: Admin aplica um ou mais filtros.
- Quando: Filtro é confirmado.
- Então: Lista exibe skeleton inline (não tela de loading completa), depois renderiza resultados filtrados. Chips dos filtros ativos aparecem acima da lista. Cada chip tem botão x funcional. "Limpar filtros" remove todos os chips e restaura a lista sem filtros.

---

**RF-004** ← RN-DA-030
Toda visualização da lista de interações pelo Admin deve ser registrada nos logs de acesso do Admin com: Admin ID, timestamp, filtros aplicados (se houver).

**Critério de aceite RF-004:**
- Dado: Admin acessa e/ou filtra a lista de interações.
- Quando: A listagem é carregada (com ou sem filtros).
- Então: Log de acesso registrado em `admin_access_logs` com Admin ID, timestamp ISO 8601 e filtros aplicados (null se nenhum).

---

**RF-005** ← RN-DA-031
O sistema deve monitorar continuamente as métricas operacionais dos agentes e disparar alertas automáticos para as seguintes condições:

| Alerta | Condição | Canal |
|---|---|---|
| Latência alta | Tempo de resposta acima do SLA por 5 minutos consecutivos | Slack + painel Admin |
| Taxa de erro elevada | > 10% das respostas com erro em 15 minutos | Slack + e-mail Admin |
| Desligamento automático | Taxa de erro > 30% em 15 minutos | Slack + e-mail Admin + painel |
| CSAT degradado | Média < 3,5/5 nas últimas 24 horas | Painel Admin + e-mail |
| Taxa de recusa alta | > 20% das respostas com recusa em 24 horas | Painel Admin |
| Consumo de processamento | > 80% do orçamento mensal | E-mail Admin |

**Critério de aceite RF-005:**
- Dado: Sistema monitora métricas em tempo real.
- Quando: Qualquer condição da tabela acima é atingida.
- Então: Alerta é disparado para os canais correspondentes dentro de 60 segundos da detecção da condição. Alerta registrado no painel Admin com: tipo de alerta, timestamp e condição que o gerou.

---

**RF-006** ← RN-DA-031
Quando múltiplos alertas são disparados simultaneamente, o sistema deve listá-los em ordem de prioridade: desligamento automático primeiro, latência alta segundo. Alertas de menor prioridade listados na sequência.

**Critério de aceite RF-006:**
- Dado: Dois ou mais alertas são disparados no mesmo ciclo de monitoramento.
- Quando: O painel Admin exibe os alertas.
- Então: Desligamento automático aparece no topo. Latência alta aparece em segundo. Demais alertas listados abaixo, por ordem de criticidade.
