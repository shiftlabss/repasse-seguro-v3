### Módulo 4 — Configuração do Agente

**RF-014** ← RN-DA-035
O Admin deve poder visualizar e ajustar o threshold de confiança para sinalização automática de interações em Configurações > Supervisão IA. Valor padrão: 80%. Range permitido: 50% a 95%.

**Critério de aceite RF-014:**
- Dado: Admin acessa Configurações > Supervisão IA.
- Quando: A tela carrega.
- Então: Threshold atual é exibido (padrão: 80% na primeira vez). Campo de input ou slider para ajuste está disponível. Range 50–95% é comunicado visualmente.

---

**RF-015** ← RN-DA-035
Ao salvar um novo threshold válido (50%–95%), o sistema deve:
1. Salvar o novo valor imediatamente.
2. Aplicar o novo threshold a todas as novas interações a partir daquele momento.
3. Exibir toast: "Nível de supervisão atualizado para [valor]%."
4. Registrar no log de auditoria: "Alterado de [anterior]% para [novo]% por [Admin] em [data/hora]."

**Critério de aceite RF-015:**
- Dado: Admin define threshold entre 50% e 95% e salva.
- Quando: O salvar é confirmado.
- Então: Toast de confirmação aparece com o novo valor. Log de auditoria registrado com os 4 campos acima. Próximas interações são sinalizadas com base no novo threshold imediatamente.

---

**RF-016** ← RN-DA-035
Ao tentar salvar threshold fora do range 50%–95%, o sistema deve exibir mensagem de erro inline: "O nível de supervisão precisa estar entre 50% e 95%. Valores fora desse intervalo podem comprometer a qualidade do atendimento." Valor inválido permanece no campo para correção. Nenhum modal é fechado.

**Critério de aceite RF-016:**
- Dado: Admin define threshold < 50% ou > 95%.
- Quando: Admin tenta salvar.
- Então: Mensagem de erro aparece inline abaixo do campo, em cor de alerta. Valor inválido permanece no campo. Nenhuma ação de salvar é executada. Nenhum modal/drawer é fechado.

### Módulo 5 — Canal Webchat (Configuração)

**RF-017** ← RN-DA-036
O sistema deve disponibilizar o webchat 24/7 desde que a API do modelo de IA esteja disponível. Quando a API estiver disponível, o chat funciona normalmente independente do horário.

**Critério de aceite RF-017:**
- Dado: API do modelo de IA está disponível.
- Quando: Usuário tenta acessar o chat em qualquer horário.
- Então: Chat abre e responde normalmente. Sem restrição de horário de funcionamento.

---

**RF-018** ← RN-DA-036
Quando a API do modelo de IA estiver indisponível, o sistema deve:
1. Exibir no chat: "O agente está temporariamente indisponível. Os cálculos de comissão e Escrow continuam disponíveis. Tente novamente em instantes."
2. Manter a Calculadora de Comissão ativa.
3. Exibir badge de status degradado (cor amarela) no ícone do FAB global antes de o usuário abrir o chat.

**Critério de aceite RF-018:**
- Dado: API do modelo de IA está indisponível.
- Quando: Usuário tenta acessar o chat.
- Então: Mensagem de indisponibilidade é exibida. Calculadora de Comissão permanece funcional. FAB global exibe badge amarelo de status degradado visível antes da abertura do chat.

---

**RF-019** ← RN-DA-036
O rate limit do webchat deve ser de 30 mensagens por hora por usuário (janela deslizante). Histórico de conversa persiste por 90 dias.

**Critério de aceite RF-019:**
- Dado: Usuário envia 30 mensagens em menos de 1 hora.
- Quando: O usuário tenta enviar a 31ª mensagem.
- Então: Sistema bloqueia a mensagem e exibe feedback de rate limit. A janela deslizante de 1 hora é calculada a partir da primeira mensagem do lote. Histórico de interações dos últimos 90 dias disponível para visualização do Admin no painel.

### Módulo 6 — Critérios de Prontidão para Lançamento

**RF-020** ← RN-DA-037
Antes da ativação do modelo de IA em produção, o sistema deve verificar se os seguintes itens estão implementados e testados:
1. **Filtro de escopo:** toda consulta de dados valida que o recurso pertence ao usuário autenticado antes de qualquer processamento.
2. **Filtro de contexto:** as informações fornecidas ao agente contêm apenas dados autorizados para o perfil do usuário.
3. **Teste de penetração:** cenário de acesso cruzado de dados (ex: Cessionário tentando ver dados do Cedente) bloqueado em 100% dos casos testados.

**Critério de aceite RF-020:**
- Dado: Equipe de engenharia solicita ativação do modelo em produção.
- Quando: Checklist de prontidão é executado.
- Então: Os 3 itens acima são verificados. Se todos aprovados → ativação autorizada. Se qualquer item falhar → ativação bloqueada até correção e reteste. Resultado do checklist registrado com timestamp e responsável.

---

**RF-021** ← RN-DA-037
Se qualquer item de isolamento de acesso (RF-020) falhar na verificação, a ativação do modelo de IA é bloqueada até correção e reteste confirmado.

**Critério de aceite RF-021:**
- Dado: Um ou mais itens do RF-020 falham na verificação.
- Quando: Checklist é executado.
- Então: Ativação do modelo não ocorre. Falha registrada com detalhes do item. Equipe notificada para correção. Reteste obrigatório antes de nova tentativa de ativação.

---

**RF-022** ← RN-DA-038
As instruções permanentes do agente devem ser aprovadas antes do lançamento, cobrindo obrigatoriamente:
1. Definição de identidade e tom do agente.
2. Lista explícita dos dados bloqueados com exemplos de recusa para cada tipo.
3. Exemplos de perguntas que devem ser recusadas com a resposta esperada.
4. Formato de resposta: toda resposta encerra com um próximo passo claro.

**Critério de aceite RF-022:**
- Dado: Equipe de produto submete instruções do agente para aprovação pré-lançamento.
- Quando: Aprovação é avaliada.
- Então: Os 4 itens acima são verificados como presentes. Aprovação concedida apenas quando todos os 4 estão cobertos.

---

**RF-023** ← RN-DA-038
Antes do lançamento, as instruções do agente devem ser testadas com no mínimo 20 perguntas adversariais que tentam extrair dados bloqueados. 100% das perguntas adversariais devem ser recusadas corretamente.

**Critério de aceite RF-023:**
- Dado: Instruções do agente estão prontas para lançamento.
- Quando: Testes adversariais são executados.
- Então: 20 perguntas adversariais executadas. 100% das perguntas resultam em recusa correta do agente. Resultado dos testes registrado. Se qualquer pergunta não for recusada → instruções revisadas e testes repetidos.

---

**RF-024** ← RN-DA-039
Antes do lançamento de qualquer agente, os seguintes componentes de supervisão devem estar implementados e testados:
1. **Registro de interações:** cada interação grava pergunta, resposta, nível de confiança e latência.
2. **Dashboard de métricas:** volume, top perguntas, taxa de recusa, CSAT e latência visíveis no painel Admin.
3. **Alerta automático de confiança:** interações com confiança abaixo do threshold sinalizadas automaticamente.
4. **Takeover manual:** Admin pode assumir conversa com registro de motivo.

**Critério de aceite RF-024:**
- Dado: Equipe de engenharia solicita autorização para lançamento do agente.
- Quando: Checklist de supervisão é executado.
- Então: Os 4 componentes acima são verificados como operacionais. Se todos operacionais → lançamento autorizado. Se qualquer componente estiver ausente → lançamento bloqueado.

---

**RF-025** ← RN-DA-039
Se qualquer componente de supervisão (RF-024) estiver ausente, o lançamento do agente é bloqueado até que o componente esteja implementado e testado.

**Critério de aceite RF-025:**
- Dado: Um ou mais componentes de supervisão do RF-024 falham na verificação.
- Quando: Checklist de lançamento é executado.
- Então: Lançamento não ocorre. Componente faltante identificado e registrado. Equipe notificada. Reteste obrigatório após implementação.
